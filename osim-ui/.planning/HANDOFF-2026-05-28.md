## 0. Strategische Vision (User-Direktive)

> **GraphObject soll ein Client-Container werden, den ich in ALLEN meinen
> Projekten weiterentwickeln will.**

Konsequenz für die Architektur:

- Der **GraphObject-Kern muss renderer-agnostisch** sein (keine harte Kopplung
  an React Flow oder ein anderes Framework). Reine TS-Bibliothek, in jedes
  Projekt einbettbar.
- Der **RF-Adapter ist eine osim-ui-spezifische Bindung** und gehört in eine
  isolierte Schicht.
- **Klassen-/Methoden-/Attribut-Namen 1:1 aus dem C++-Original** übernehmen
  (siehe Memory `feedback-consult-original-code`). Keine erfundenen
  Vereinfachungen oder eingedeutschten Code-Identifier.

   konfigurierbare Shapes auslagern.
5. **`OGPositionList`** — Listen-Layout-Position (für `OGraphList`-Konsumenten).
6. **`GObjOSimDlp`** — OSim-spezifischer Durchlaufplan-GObj-Subtyp.

## 4. React Flow — was es leistet und wo seine Grenze ist

**Version:** `@xyflow/react@^12.10.2` (in `portal/package.json`).

### 4.1 Was React Flow LEISTET (osim-ui nutzt diese Features)

| Feature | RF-API | Genutzt in |
|---|---|---|
| Node-/Edge-Rendering mit absoluter Positionierung | `ReactFlow nodes/edges` | `GraphFlowCanvas`, `OsimNode` |
| Pan/Zoom-Viewport | `ReactFlow viewport`, `useReactFlow().fitView` | `GraphFlowCanvas` (Welle G10 fitView-on-revision) |
| Drag-Interaktion (visuell + Drop-Callbacks) | `onNodeDragStop` | `interactions.ts::onNodeDragStop` |
| Verbindungs-Handles (Drag zwischen Knoten) | `Handle`, `onConnect` | `OsimNode` (4 Handles N/S/E/W), `interactions.ts::onConnect` |
| Selektion (single + Box) | `onNodesDelete`, `onEdgesDelete` | `interactions.ts` |
| Delete-Taste | `deleteKeyCode` | `GraphFlowCanvas` (Welle G8) |
| Hintergrund-Grid via ViewportPortal | `ViewportPortal` | `GridBackground` (Welle G3) |
| Custom-Node-Komponenten | `nodeTypes`, `React.memo` | `OsimNode`, `OsimEdgeBox` |
| Custom-Edge-Routing | `edgeTypes` (`step`, `smoothstep`) | `view-adapter::edgeTypeFor` |
| Edge-Pfeil-Spitzen | `markerEnd` | `view-adapter` |

### 4.2 Was React Flow NICHT leistet (= GraphObject-Zuständigkeit)

| Verantwortung | Hier: GraphObject-Klasse | osim-ui-Stelle |
|---|---|---|
| Domänen-Datenmodell (Knoten/Kanten-Semantik) | `GObject`, `GObjLink`, `GLink` | `foundation/G*.ts` |
| Grid-basiertes Layout (Spalten/Zeilen, Zell-Position `m_pntRaster`) | `OGraphGrid` | `OGraphGrid.ts`, `wire-to-grid.ts` |
| Dynamisches Spalten-Resize (Max-Width pro Spalte) | `OGraphGrid::GetSize/SetPosition` | `OGraphGrid::computeSizes/applyPositions` (Welle G7) |
| Nested Sub-Views (Alt-/Plan-Hierarchie) | `GObjSub` (synth.) / `GObjElements`? | `GObjSub.ts` (Welle G6) |
| Region Check (Hit-Test in Grid-Koordinaten) | `OGraphGrid::GetGridAtPoint`, `GetNextFreeGridPlace` | `OGraphGrid.ts`, `interactions.ts` |
| Multi-Layer-Drawing (Hintergrund / Inhalt / Vordergrund / Overlay) | `GObject::Draw/DrawHelpers/DrawText` | ❌ **NICHT IM PORT** — RF erledigt z-Index per DOM, aber die 4-Layer-Semantik fehlt. |
| Phantom-Preview (Drag-Ghost-Modell) | `GObject::DrawPhantom`? | ❌ **NICHT IM PORT** — RF zeigt nur eigenen Drag-State; kein Phantom-Modell. |
| In-Place-Editor | `GObjCEdit` | ❌ **NICHT IM PORT** (siehe §3.1 Punkt 2). |
| Insert/Remove/Move-Mechanik mit Layout-Update | `OGraphGrid::GOIns/GORemove/GOMove` | `OGraphGrid::GOIns`, `interactions.ts::onNodeDragStop` (Welle G8). |
| Listen-Layout (linear) | `OGraphList` | `OGraphList.ts` (leicht genutzt). |

### 4.3 Architektur-Vertrag (heute)

```
       Wire (JSON)
            ↓ wire-to-grid.ts            [PURE]
        OGraphGrid (GObject-Hierarchie)
            ↓ view-adapter.ts            [RF-Adapter]
        React-Flow Nodes/Edges
            ↓ GraphFlowCanvas.tsx        [RF-Adapter]
        DOM-Rendering durch React Flow
            ↑ interactions.ts            [RF-Adapter]
        Wire-Mutation (createObject/patchObject/appendSubRef)
```

`OGraphGrid` ist Single-Source-of-Truth für Layout/Struktur; RF ist reiner
Renderer + Event-Quelle. **Keine RF-Datenstruktur fließt in das
GraphObject-Modell zurück.**

## 5. Was zu tun ist, um `GraphObject` zum portablen Client-Container zu
machen

### 5.1 Sofort (Aufräumen, niedriges Risiko)

- [ ] **`graph/core/` löschen** (toter Code, Phase-1-Minimalport ohne
      Konsumenten; nur `core/__tests__/graph-builder.spec.ts` als
      Test-Altlast).
- [ ] **`PRessMengeMatrixViewer` auf `MatrixGrid`-Foundation migrieren**
      (eliminiert `matrix-common.tsx`).
- [ ] **`GObjSub` ↔ C++ `GObjElements` abgleichen** (Klassen-Name, Methoden)
      und ggf. umbenennen/refaktorieren, damit der Foundation-Code 1:1 zum
      Original ist.

### 5.2 Foundation vervollständigen (Lücken aus §3.1)

- [ ] **`GObjCEdit`** portieren — In-Place-Editor-Knoten (Beschriftungs-Edit
      direkt am Graph).
- [ ] **`OGGridAlt`** portieren — Alternativ-Logik-Grid für
      `PDpKnAlternativ`/Splits.
- [ ] **`GObjSquare`, `GObjRect`** als eigene Klassen, statt SVG-Polygon in
      `OsimNode.tsx` hartzucodieren.
- [ ] **`GObjOSimDlp`** — OSim-spezifischer Durchlaufplan-GObj-Subtyp.
- [ ] **`OGPositionList`** — Listen-Layout-Position für `OGraphList`.

### 5.3 Multi-Layer-Drawing + Phantom-Modell

- [ ] **4-Layer-Draw-API** auf `GObject` portieren (`Draw`, `DrawHelpers`,
      `DrawText`, `DrawOverlay`) — als Renderer-agnostische Konventionen, die
      der RF-Adapter dann auf z-Index/SVG-Layer abbildet.
- [ ] **Phantom-Preview-Modell** (`GObject::DrawPhantom` o.ä.) — ein
      Vorschau-Objekt während Drag, das vom Renderer angezeigt wird.

### 5.4 Renderer-Agnostik finalisieren

- [ ] **Foundation-Kern als eigenes Paket** auslagern (`packages/graphobject/`
      o.ä. — workspaces-fähig), damit es projektübergreifend einbettbar ist.
      Der RF-Adapter (`GraphFlowCanvas`, `view-adapter`, `interactions`,
      `OsimNode`, `GridBackground`) wandert in eine separate
      `graphobject-react-flow`-Schicht.
- [ ] **Alternative Renderer-Adapter** prototypisch (Canvas, SVG, …),
      um die Renderer-Unabhängigkeit zu validieren.
