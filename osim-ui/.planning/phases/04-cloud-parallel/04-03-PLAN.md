<interfaces>
osim-engine Monorepo-Layout (siehe System-Reminder: "Engine liegt im selben Monorepo
`engine/`, Submodule entfällt!"):
- Engine-Code in `engine/` (relativ zum osim-ui-Repo-Root).
- `engine/pyproject.toml` definiert Package `osim_engine`.
- Im Worker-Image als editable install: `uv pip install -e ./engine`.

Phase-2-Annahmen (vor Phase 4 fertig):
- `app/services/orchestrator.py` startet lokal Subprocess `python -m worker ...` mit
  Args. Phase 4 ersetzt diesen Aufruf durch Cloud Tasks (Plan 04-04).
- `worker/`-Modul existiert ggf. schon als Phase-2-Variante. Phase 4 erweitert es um
  Cloud-Run-Job-Fähigkeit und Pub/Sub-Anbindung (statt In-Process-Queue).

Cloud-Run-Job-Payload-Format (von Cloud Tasks via HTTP-Trigger ODER via env-Vars
bei direct-invoke):
```json
{
  "run_id": "uuid",
  "multi_run_id": "uuid|null",
  "seed_index": 0,
  "tenant_id": "tenant_xyz",
  "model_uri": "gs://osim-ui-staging-data/models/{tenant}/{model_id}/model.json",
  "config": {
    "seed": 42,
    "start_date": "2026-01-01T00:00:00Z",
    "end_date": "2026-01-31T00:00:00Z",
    "period_len": 3600
  }
}
```

Engine-API (aus .planning/research/osim-engine-api.md):
- `PSimulator(...)`, `PSimulator.eventbus` (EventBus), `PSimulator.start()`.
- `EventBus.subscribe(topic_pattern, callback)` für Recorder-Pattern.
- Reproduzierbarkeitsvertrag: (seed, start_date, end_date, period_len) MÜSSEN in
  exakter Reihenfolge gesetzt werden VOR PSimulator.start().

Cloud Run Job spezifika:
- Max execution time pro Task: 60min (default), kann via `--task-timeout` auf bis zu
  24h gesetzt werden. Default Phase 4: 30min, override pro Tenant in Plan 04-04.
- env-Vars für Job: K_SERVICE, CLOUD_RUN_TASK_INDEX, CLOUD_RUN_TASK_COUNT,
  CLOUD_RUN_TASK_ATTEMPT, CLOUD_RUN_EXECUTION.
- Secrets als Volume oder env-Vars; Phase 4 nutzt env-Vars (einfacher).

    worker/runner.py — SimRunner:
    - `__init__(payload: WorkerPayload, sink: EventSink, snapshot: SnapshotWriter)`.
    - `async run() -> None`:
      1. publish status="running" (status_topic, run_id, started_at).
      2. Modell aus GCS laden (storage.read_model(model_uri)).
      3. PSimulator instanziieren, Reproduzierbarkeitsvertrag setzen
         (seed, start_date, end_date, period_len) BEVOR weiterer Setup.
      4. EventBus-Subscriber registrieren: subscribe("*", callback). Callback wandelt
         in SimEvent und macht sowohl sink.publish() als auch snapshot.append().
      5. PSimulator.start() in einem ThreadPoolExecutor (Engine ist sync und
         CPU-bound, soll Event-Loop nicht blockieren).
      6. Bei Success: publish status="succeeded" mit final_sim_time, event_count.
      7. Bei Exception: publish status="failed" mit error_type, error_message.
      8. Bei SIGTERM (Cloud Run Job kill): publish status="cancelled", snapshot.close().
      9. finally: snapshot.close() (final flush).
    
    worker/snapshot_writer.py — SnapshotWriter:
    - `__init__(gcs_uri, flush_interval_seconds)`.
    - `append(event: SimEvent)`: in In-Memory-Buffer (list).
    - Background-Task: alle N Sekunden flush(): schreibt Buffer als JSONL-Append
      nach GCS (compose-API: vorhandenes Object + neuer Chunk zu neuem Object;
      alternative: pro Flush ein neues File `events-{batch_index:06d}.jsonl`).
    - `async close()`: cancel background task, final flush.
    - Lokal: alternativer LocalSnapshotWriter mit Datei-Append (für Tests und Dev).
    
    worker/__main__.py:
    - Liest Payload aus env-Var `WORKER_PAYLOAD` (JSON-String) ODER aus stdin
      (für lokales Subprocess-Mode aus Phase 2).
    - SignalHandler für SIGTERM (Cloud Run Job graceful shutdown).
    - Setzt SimRunner zusammen, ruft asyncio.run(runner.run()).
    - Exit-Code 0 für Success, 1 für Failure (Cloud Run Job Retry-Logik).
    
    app/services/storage_gcs.py: GCS-Backend für Storage-Abstraktion aus Phase 1.
    Methoden: `read_model(uri)`, `write_snapshot(uri, jsonl)`, `compose(uris, target_uri)`.
    Authentifizierung über Default Service Account Credentials (im Cloud Run Job
    automatisch verfügbar).

    - test_snapshot_writer: fake-gcs-server oder MinIO-Mock, freezgun für Time-Control.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker-Container → Engine-Code | Trust, da Monorepo-Build |
| Worker → GCS (Modell-Download) | Auth via osim-worker-sa, tenant-isoliert per Pfad-Präfix |
| Worker → Pub/Sub | publishen, kein subscribe |
| Cloud Tasks → Worker-Job | OIDC-Token-Auth (siehe Plan 04-04) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-12 | T (Tampering) | model_uri-Manipulation | mitigate | API validiert vor Enqueue, dass model_uri zum tenant_id passt (Pfad-Präfix-Check) |
| T-04-13 | I | Worker leakt Events anderer Tenants | mitigate | run_id-Attribute auf jeder Publish; Subscriber filtert; Tenant-Trennung über run_id |
| T-04-14 | D | Endlosschleife in Engine | mitigate | worker_timeout_seconds als Hard-Limit; Cloud Run Job killt nach Timeout |
| T-04-15 | E | Worker eskaliert via Service-Account | mitigate | osim-worker-sa hat keine roles/iam.serviceAccountAdmin, nur Lesen/Schreiben in spezifischen Buckets/Topics |
| T-04-16 | I | Modell-Datei zu groß → OOM | mitigate | Streaming-Read aus GCS, Memory-Limit Cloud Run 4Gi (klare Fehlermeldung bei OOM) |
| T-04-17 | R | Audit: wer hat welchen Run gestartet | accept | run-Tabelle in DB hat created_by + tenant_id; Cloud Run Execution-ID wird mit run-Row verlinkt (Plan 04-04) |
</threat_model>
