<objective>
Multi-Run-Submission ("N Seeds parallel") und KPI-Aggregation über alle Sub-Runs.
User submittet ein Modell mit `repeats=10` → 10 unabhängige Cloud Run Jobs starten
parallel (innerhalb der Tenant-Quota) → Aggregations-Dashboard zeigt Mean/Median/CI
über alle Seeds.

Purpose: Statistisch belastbare Aussagen aus stochastischen Simulationen, klassisches
Monte-Carlo-Vorgehen. Kernanwendung für Beratungsprojekte (What-if-Studien) und
Klausur-Szenarien (Reproduzierbarkeitsgarantie über Seed-Sets).
Output: Multi-Run-DB-Modell, Service-Schicht, REST-API, Frontend-Komponenten.
</objective>

DB-Schema (zusätzlich zu Phase-2-Tabellen):
- public.multi_runs (cross-tenant? NEIN — multi_runs lebt PRO Tenant-Schema, weil
  sie eine User-Aktion innerhalb eines Tenants sind):
  * id UUID PK
  * tenant_id VARCHAR NOT NULL (Redundanz für Cross-Schema-Queries, falls nötig)
  * model_id UUID NOT NULL → models(id)
  * total_repeats INT NOT NULL
  * seed_strategy VARCHAR NOT NULL CHECK (seed_strategy IN ('sequential', 'random'))
  * base_seed BIGINT NOT NULL (bei sequential: seeds = base_seed + seed_index;
    bei random: pro Sub-Run aus random.SystemRandom)
  * config_snapshot JSONB NOT NULL (Sim-Config geteilt von allen Sub-Runs ohne seed)
  * status VARCHAR NOT NULL CHECK (status IN ('queued', 'running', 'partial', 'completed', 'failed'))
  * created_by VARCHAR NOT NULL
  * created_at TIMESTAMPTZ DEFAULT now()
  * completed_at TIMESTAMPTZ NULL
- runs-Tabelle hat schon multi_run_id und seed_index (in Plan 04-04 hinzugefügt).

POST /api/v1/runs Erweiterung:
- Body: { model_id, config: {seed?, start_date, end_date, period_len},
  repeats?: int = 1, seed_strategy?: "sequential" | "random" = "sequential" }
- Wenn repeats == 1: alter Pfad (single-run), kein multi_run-Eintrag.
- Wenn repeats > 1: legt multi_run + N runs an, ruft orchestrator.submit() für jeden.

POST /api/v1/multi-runs (alternative, explizite Route):
- Body: { model_id, config, repeats: int, seed_strategy }.
- Vorteil: klare Semantik, einfacher zu validieren.
- Beide Routen werden angeboten (User-Convenience).

GET /api/v1/multi-runs/{id}:
- Response: { multi_run: {...}, sub_runs: [{run_id, seed_index, seed, status, kpis}],
  aggregation: { kpi_name: { mean, median, std, min, max, p05, p95, ci95: [lo, hi],
  n_succeeded, n_failed } } }.

KPI-Quellen:
- Engine liefert pro Run KPIs als Teil des EventBus-Streams (Topic z.B.
  "kpi.final.{name}" mit Wert). Phase 3 sammelt die im API.
- Aggregations-Service liest aus runs-Tabelle Spalte `final_kpis JSONB`
  (von Phase 3 gefüllt; falls nicht da: aus events.jsonl-Snapshot parsen).
- Phase-4-Annahme: runs.final_kpis existiert als JSONB-Spalte. Falls nicht, in
  Migration ergänzen.

Aggregations-Algorithmik (klassisch):
- mean = sum(x_i) / n
- std = sqrt(sum((x_i - mean)²) / (n-1))  # sample-std
- ci95 = mean ± 1.96 * std / sqrt(n)  # Normalapprox; bei n < 30 t-Verteilung
  (statsmodels oder scipy.stats.t.interval(0.95, n-1, mean, std/sqrt(n)))
- Histogramm: 20 Bins zwischen min und max.

Reproduzierbarkeitsvertrag (Engine):
- (seed, start_date, end_date, period_len) muss persistent sein.
- Bei seed_strategy=sequential: seed_i = base_seed + i. base_seed wird beim Anlegen
  gesetzt; falls user keinen vorgibt: random.SystemRandom.randint(0, 2**31).
- Bei seed_strategy=random: seed_i = SystemRandom.randint() pro i. base_seed=0 wird
  als Marker gespeichert.
- Beide Strategien legen alle N Seeds persistent in runs.config.seed ab.

Frontend:
- TanStack Router Route `/multi-runs/new`: Form mit Modell-Picker, Repeat-Slider
  (1-50, Default 10), Strategy-Dropdown, Sim-Config-Editor.
- Route `/multi-runs/{id}`: Dashboard mit:
  * Status-Banner (queued: X/N, running: X/N, succeeded: X/N, failed: X/N).
  * KPI-Karten: für die 4-6 wichtigsten KPIs Mean ± CI.
  * Histogramm-Panel pro KPI (Recharts BarChart).
  * Run-Grid: Tabelle mit allen Sub-Runs (seed_index, seed, status, key-KPIs),
    Click → Single-Run-Detail.
  * Auto-Refetch alle 3s während status != completed.
