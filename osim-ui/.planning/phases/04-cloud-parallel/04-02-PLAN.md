Live-Streaming von Engine-Events vom In-Process-Modell (Phase 3) auf Pub/Sub umstellen,
damit Worker und API über mehrere Replicas hinweg laufen können. Phase 3 bleibt für
lokale Entwicklung erhalten (Feature-Flag `EVENT_BACKEND=inprocess`).

Purpose: Single-Host-Architektur aus Phase 3 skaliert nicht über Cloud Run Replicas.
Pub/Sub entkoppelt Worker-Pod und API-Pod und macht horizontales Skalieren möglich.
Output: PubSubEventSink mit Backpressure und Resync, neuer WS-Endpoint, Terraform-
Modul für Topics und Subscriptions, Tests.
</objective>

Pub/Sub-Topic-Schema:
- `osim-runs-events-{env}`: ein Topic für ALLE Runs. Message-Attribute: `run_id`,
  `seed_index`, `topic`, `sim_time`. Body: JSON-Payload des Engine-Events.
- `osim-runs-status-{env}`: separates Topic für Lifecycle-Events (queued, running,
  succeeded, failed, cancelled). Selteneres Volumen, separater Subscriber.
- Subscriptions: KEINE shared Subscription. Pro WS-Client wird eine dynamische
  Subscription `osim-ws-{run_id}-{ws_session_id}` angelegt mit Filter
  `attributes.run_id = "{run_id}"`, retention 10min, ack_deadline 60s,
  enable_message_ordering=false.
- Lifecycle: Subscription wird bei WS-Disconnect automatisch gelöscht (asynccontextmanager).

Backpressure-Strategie:
- WsSubscriber hat asyncio.Queue(maxsize=500). Bei put_nowait()-OverflowError:
  Drop-oldest + Sample-Counter incrementieren, alle 50 Drops ein Sample-Event an
  Client mit `{type: "sample", dropped: N, since_sim_time: X}`.

Resync-Strategie:
- Worker schreibt parallel zum Pub/Sub-Publish einen JSONL-Snapshot nach
  `gs://osim-ui-{env}-data/traces/{run_id}/events.jsonl` (alle 60s flushen).
- Bei WS-Reconnect mit Query-Param `?since_sim_time=X.Y`: API liest aus GCS-Snapshot
  alle Events mit sim_time >= X.Y, streamt sie zuerst, schaltet dann auf Pub/Sub-Live um.

Diskuss-Punkt aus PRELIMINARY-PLAN (Pub/Sub-Sampling bei vielen Events):
- Default: KEIN serverseitiges Sampling. Backpressure greift erst clientseitig.
- Optional via Config: `EVENT_SAMPLING_RATE=1.0` (alles) bis `0.1` (jeder 10.). Wenn
  <1.0, Sampling im Worker vor publish().

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-08 | D | Subscription-Leak bei API-Crash | mitigate | Lifecycle-Manager: Subscriptions mit message_retention=600s und Worker, der nach 1h Subscriptions ohne ack löscht (separate Garbage-Job in Plan 04-04) |
