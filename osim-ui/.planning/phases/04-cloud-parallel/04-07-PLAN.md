Stress-Test-Setup:
- Locust mit 10 simulierten Tenant-Accounts (Firebase-Test-User pre-seeded).
- Jeder User submittet 5 Multi-Runs mit repeats=1 (single-run) oder 1 Multi-Run mit
  repeats=5 — beide Varianten messen.
- Tenant-Quota=5 (Default), also: 5 Runs gleichzeitig pro Tenant, 50 insgesamt.
- Sim-Modell: kleinstes Dummy-Modell (Phase 1: Dummy.otx → JSON konvertiert in
  Phase 2). Erwartete Sim-Dauer pro Run: 10-30 Sekunden.
- Erwartung: 50 Runs in unter 60 Minuten = Mittel pro Run < 72 Sekunden inkl.
  Cold-Start + Queue-Time + Sim-Dauer.

Failure-Mode-Tests:
- Worker-OOM: synthetisches Modell, das in Engine OOM erzwingt (large in-memory
  Datenstruktur). Erwartung: Cloud Run Job exit-code 137, status=failed,
  error_message enthält "OOM" oder "memory".
- Worker-Timeout: Modell mit künstlicher infinite-loop (Test-Engine-Variante oder
  Modell mit extrem vielen Perioden). Erwartung: nach 1800s SIGTERM, status=cancelled,
  klare Meldung im UI.
- Pub/Sub-Backpressure: hochfrequenter Event-Strom (Test-Modell). Erwartung: WS-
  Client sieht sample-Marker, kein crash.

Cost-Monitoring:
- google_billing_budget: Budget-Name "osim-ui-{env}-monthly", 
  Beträge: 50 EUR staging, 200 EUR prod (Defaults, anpassbar).
  Notifications bei 50%, 80%, 100%, 120%. Empfänger: jwfischer69@gmail.com.
- google_monitoring_notification_channel: email für oben + optional Pub/Sub für
  spätere Integration in tbx_stzrim.
- google_monitoring_dashboard:
  * Cloud Run API: Request Count, Latency p50/p95/p99, Error Rate.
  * Cloud Run Worker Job: Executions Count, Execution Duration, Failed Executions.
  * Pub/Sub: Publish Rate, Subscriber Lag, Undelivered Messages.
  * Cloud SQL: CPU, Memory, Active Connections, Disk Usage.
  * Cloud Tasks: Queue Depth, Dispatch Rate.
