<objective>
Den Orchestrator von lokalem Subprocess-Spawning auf Cloud Tasks umstellen. Cloud
Tasks dispatcht HTTP-Requests an den Cloud Run Job aus Plan 04-03. Pro-Tenant-Quota
wird vor Enqueue geprüft, Per-Run-Timeout durch Cloud Run Job's max_duration erzwungen.

Purpose: Skalierbares, persistentes Queueing mit Retry-Policy und Rate-Limiting.
Cloud Run Jobs können nicht direkt aus Cloud Tasks invoked werden mit "fire-and-forget",
deshalb läuft der Dispatch über HTTP-Trigger auf den Job's "executions:run"-Endpoint.
Output: CloudTasksOrchestrator, TenantQuota-Service mit DB-Tabelle, Terraform-Modul,
Subscription-Garbage-Collector.
</objective>

<interfaces>
Phase-2-Annahme (Orchestrator existiert):
- `app/services/orchestrator.py` aus Phase 2 mit submit(run_id, model_uri, config) →
  startet Subprocess. Phase 4 generalisiert zum Protocol.

Cloud Tasks → Cloud Run Job (HTTP-Variante):
- Job hat HTTP-Endpoint:
  `POST https://{region}-run.googleapis.com/apis/run.googleapis.com/v1/namespaces/{project}/jobs/osim-worker-{env}:run`
- Body: `{"overrides": {"containerOverrides": [{"env": [{"name": "WORKER_PAYLOAD", "value": "{...json...}"}]}]}}`.
- Auth: OIDC-Token, signiert mit Cloud Tasks Service-Account (osim-tasks-sa, NEU in
  diesem Plan, oder Reuse osim-build-sa). Audience = Run-API.
- Per-Tenant-Cloud-Tasks-Queue ist möglich (eine Queue pro Tenant), wird aber NICHT
  in Phase 4 gemacht (ein gemeinsamer Queue mit Tenant-Filter in Code).

Tenant-Quota-Modell:
- Tabelle `tenant_quotas` (im public-Schema, nicht im Tenant-Schema, weil Quota
  ein cross-tenant-Concept ist):
  * tenant_id (PK)
  * max_concurrent_runs (int, default 5)
  * default_worker_timeout_s (int, default 1800)
  * max_runs_per_day (int, default 100; nicht erzwungen in Phase 4, vorbereitet)
- check_and_reserve(tenant_id):
  1. SELECT count(*) FROM runs WHERE tenant_id=$1 AND status IN ('queued', 'running').
  2. Vergleich mit max_concurrent_runs aus tenant_quotas (Default 5 wenn kein Eintrag).
  3. Wenn Quota erreicht: return False, sonst return True (kein DB-Update — Reservierung
     wird durch run-Insert mit status='queued' atomisch in derselben Transaction gemacht).

Subscription-Garbage:
- Hintergrund-Job (FastAPI background task oder eigener Cloud Run Job mit Cloud
  Scheduler, Phase 4 nutzt FastAPI background als einfache Variante):
  alle 5 Minuten: list_subscriptions(project), für jede `osim-ws-*`-Subscription:
  prüfe last-ack-time. Wenn > 1h: delete_subscription.

OFFENE FRAGE (aus PRELIMINARY-PLAN, Punkt 4):
**Pub/Sub-Sampling bei vielen Events:** schon in Plan 04-02 beantwortet (Default 1.0,
Notbremse via EVENT_SAMPLING_RATE).

Phase-2-Kompatibilität:
- Lokales Dev: ORCHESTRATOR_BACKEND=local nutzt weiterhin Subprocess.
- Tests: orchestrator_cloudtasks gegen Cloud-Tasks-Emulator (es gibt keinen offiziellen,
  daher Mock via google-cloud-tasks-Client mit responses-Lib).

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-18 | D | Tenant überflutet Queue | mitigate | TenantQuotaService vor Enqueue + Cloud-Tasks-rate_limits als Backstop |
| T-04-19 | S (Spoofing) | gefälschter Cloud-Tasks-Caller | mitigate | OIDC-Token-Validation auf Job-Endpoint durch GCP-Plattform; Job nur via run.invoker erreichbar |
