  <name>Task 1: LiveWsClient mit Auto-Reconnect</name>
  <files>
    portal/src/graph/live/types.ts,

  <behavior>
    - connect(url, token): öffnet `new WebSocket(url + '?token=' + token)` und setzt status='connecting'.
    - onopen → status='connected', backoffMs=250.
    - onmessage(JSON) → callback(event).
    - onclose ohne explicit disconnect → status='reconnecting', setTimeout(reconnect, backoffMs), backoffMs = min(backoffMs * 2, 5000).
    - Nach 10 erfolglosen Reconnect-Versuchen → status='disconnected', stop.
    - disconnect() → status='disconnected', cancel pending reconnects, ws.close(1000).
    - statusChange-Callback wird bei jedem Status-Übergang gerufen.
    - Erstes Reconnect-Attempt erfolgt ≤3s nach erstem onclose (Akzeptanzkriterium 9).
  </behavior>
  <action>
    `types.ts`: Definiere `LiveEvent`, `LiveStatus = 'idle'|'connecting'|'connected'|'reconnecting'|'disconnected'|'failed'`, `LiveWsCallbacks = {onEvent, onStatusChange, onError}`.

    `ws-client.ts`: Klasse `LiveWsClient` mit private state (ws, status, attempts, backoffMs, reconnectTimer). Verwendet `globalThis.WebSocket` (in Tests mit `vi.stubGlobal('WebSocket', MockWS)` ersetzbar).

    Tests mit Mock-WebSocket: open → onStatusChange('connected'); empfangene Message wird an onEvent durchgereicht; bei close-Event ohne disconnect wird neuer WS innerhalb 250-500ms aufgemacht; nach 10 fails status='failed'; expliziter disconnect verhindert weitere Reconnects.

    Timer-Tests verwenden `vi.useFakeTimers()`.
  </action>
  <verify>

  </verify>

  <behavior>
    - EventRouter.registerNode(id, gobj) speichert Map id→gobj.
    - EventRouter.routeEvent({topic, data}) ruft gobj.onSimEvent(topic, data) für den/die betroffenen Knoten:
      • `proz.bearbeit.start`/`proz.bearbeit.ende` → data.node_id
      • `plan.ausloesen`/`plan.beendet` → data.plan_id (Plan-Knoten)
      • `sim.*` / `ptk.*` → broadcast an alle registrierten Knoten

    - KpiAggregator.feed(event) aktualisiert:
      • simTime: aus event.sim_time
      • completedPlans: increments bei `plan.beendet`
      • machineUtilization: Map<ressource_id, busyPercent> aus `proz.bearbeit.start/ende` mit Tracking offener Zeitfenster pro Ressource
    - KpiAggregator.snapshot() liefert `{simTime, completedPlans, machineUtilization: Array<{id, busyPercent, timestamp}>}` als Recharts-kompatible Datapoint-Arrays.
    - KpiAggregator.onSnapshot(cb) wird mit Throttle ≤500ms gerufen (1s-Update-Frequenz garantiert).
  </behavior>
  <action>
    `event-router.ts`: Klasse `EventRouter` mit Map. Topic-Dispatch via Switch über Topic-Prefix; Unbekannte Topics nicht droppen — als `onUnknownTopic`-Callback weiterleiten (für Plan 05 / debugging).

    `kpi-aggregator.ts`: Klasse `KpiAggregator`. Maschinen-Auslastung-Tracking: pro `ressource_id` ein State `{busySince: number|null, totalBusyTime: number, lastObservedAt: number}`. Auf `proz.bearbeit.ende`: totalBusyTime += sim_time - busySince. busyPercent = totalBusyTime / (current_sim_time - first_observed_time).

    Throttle: einfacher zeitbasierter Timer (`setTimeout`, last-write-wins) — nicht requestAnimationFrame (Aggregator ist React-frei).

    Tests: Router routes proz.bearbeit.start → richtige gobj.onSimEvent; Aggregator nach Sequence `start(node=A,res=M1,t=100) → ende(node=A,res=M1,t=600) → t=1000-snapshot` liefert M1.busyPercent ≈ 500/1000 = 50%.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Firebase-JWT im URL-Query | Browser-WebSocket-API zwingt zu Query-Param-Auth (kein Custom-Header). Token ist short-lived. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Spoofing | Token wird per Query-Param übertragen | accept | Akzeptiert via Phase-3-Plan-02 T-03-07 (kurzlebige Firebase-JWTs); im Frontend wird Token vor connect via `await user.getIdToken(true)` neu geholt um Token-Expiry-Reconnect-Loops zu vermeiden. |
| T-03-11 | DoS | Server sendet Million Events → Frontend friert ein | mitigate | rAF-Batching im Hook + Throttle ≤500ms im Aggregator. EventRouter ist O(1)-Lookup. Stress-Test in Plan 03-06. |
| T-03-12 | Info Disclosure | Unbekannte Topics werden geloggt → könnten PII enthalten | accept | OSim-Domänen-Topics enthalten keine PII (technische Node-IDs). console.warn wird in Produktion via logger-Wrapper unterdrückt. |
</threat_model>
