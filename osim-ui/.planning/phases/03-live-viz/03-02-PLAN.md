  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → API WebSocket | Untrusted client; Token im Query-Param ist sichtbar in Server-Logs (bekanntes Risiko) |
| Worker-Subprozess → API | API liest stdout — Worker ist Same-Trust (eigener Subprozess), aber JSON-Parse muss robust sein |
| API-Process zwischen Tenants | Mehrere Tenants teilen sich denselben LiveBus-Singleton |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Spoofing | WebSocket-Auth via `?token=` | mitigate | `verify_ws_token` ruft Firebase-Verifikation (existing in Phase 1); ungültige Tokens → close 1008. |
| T-03-05 | Info Disclosure | Tenant-A sieht Events von Tenant-B | mitigate | Mandatory `registry.get(run_id).tenant_id == ctx.tenant_id`-Check vor `ws.accept()`. Test in tests/api/test_runs_live_ws.py::test_foreign_tenant_rejected. |
| T-03-06 | DoS | Ein User öffnet 1000 WebSockets zum selben run_id | accept | Phase 3 single-host; explizit out-of-scope für Phase 3. Wird in Phase 4 via Per-Tenant-Quota mitigiert (siehe ROADMAP Phase 4 SC 5). |
| T-03-07 | Info Disclosure | JWT im Query-Param landet in Access-Logs | accept | Branchen-Standard für WS-Auth; Mitigation = nur kurzlebige Firebase-JWTs (Default 1h). Dokumentiert in docs/PHASE-3-GRAPH-OBJECT.md. |
| T-03-08 | Tampering | Worker emittiert manipuliertes JSON | mitigate | Bridge parsed mit try/except, skipped invalide Zeilen, loggt structlog-Warning. |
| T-03-09 | Repudiation | Run-Status-Übergänge nicht auditiert | accept | Phase 3 nicht im Audit-Scope (Beratungs-Audit-Log ist Backlog laut ROADMAP). |
</threat_model>
