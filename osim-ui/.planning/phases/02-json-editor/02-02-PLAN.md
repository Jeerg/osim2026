```python
# app/services/orchestrator.py
class RunStatus(str, Enum):
    QUEUED = "queued"
    RUNNING = "running"
    SUCCEEDED = "succeeded"
    FAILED = "failed"
    TIMEOUT = "timeout"

@dataclass
class RunConfig:
    seed: int
    periods: int
    timeout_seconds: int = 600

@dataclass
class RunSnapshot:
    run_id: UUID
    status: RunStatus
    started_at: datetime | None
    finished_at: datetime | None
    summary: dict | None
    error: str | None
    trace_uri: str | None

class RunOrchestrator:
    def submit(self, tenant_id: str, model_id: UUID, config: RunConfig) -> UUID: ...
    async def get_status(self, run_id: UUID) -> RunSnapshot: ...
    async def shutdown(self) -> None: ...
```

```python
# app/api/v1/runs.py — Pydantic-Schemas
class RunSubmitRequest(BaseModel):
    model_id: UUID
    seed: int = Field(ge=0, le=2**31 - 1)
    periods: int = Field(ge=1, le=365)
    timeout_seconds: int = Field(default=600, ge=10, le=3600)

class RunResponse(BaseModel):
    run_id: UUID
    status: RunStatus
    started_at: datetime | None
    finished_at: datetime | None
    summary: dict | None
    error: str | None

class TraceResponse(BaseModel):
    run_id: UUID
    trace_url: str
    expires_at: datetime
    size_bytes: int
```

  <files>app/services/orchestrator.py, app/services/storage.py, app/core/lifespan.py, app/main.py, tests/services/test_orchestrator.py</files>
  <behavior>
    - `RunOrchestrator.submit(...)`:
      * Erzeugt `run_id = uuid4()`, schreibt `runs`-Row mit Status `queued`.
      * Materialisiert das Modell-Artefakt aus Storage in einen temporaeren Pfad (oder uebergibt direkt eine signed URL als `file://`-Pfad fuer Local-Backend).
      * Reservoir-frei: dispatcht via `pool.apply_async(_run_subprocess_blocking, args=(...,), callback=_on_done)`. Die "Pool"-Funktion startet INTERN einen subprocess.Popen — der Pool dient nur als Scheduling-Layer mit max-concurrent-Begrenzung. Begruendung: native multiprocessing-Worker importieren die Engine -> `s_verteil`-Singleton wuerde pro Worker geteilt, aber nicht pro Submit isoliert; Popen-pro-Run garantiert frische LCG-Initialisierung.
      * Callback aktualisiert `runs.status` -> `succeeded`/`failed`/`timeout` und legt `run_artifacts`-Eintraege an.
      * Status-Transitionen: queued -> running (sobald Popen.pid bekannt) -> {succeeded|failed|timeout}.
    - `get_status(run_id)` liefert frischen Snapshot aus DB.
    - `shutdown()` schliesst den Pool, terminiert offene Subprocesses (`Popen.terminate()` mit Grace-Period 5s, dann `kill()`).
    - Pool wird im FastAPI-Lifespan-Handler angelegt (NICHT modul-global — Reload-Mode-Risk!). Pool-Size aus Config (Default 2 auf Dev-Maschine).
    - `app/services/storage.py` mit minimaler API: `class Storage: def put(self, key: str, data: bytes) -> str (uri); def get_local_path(self, uri: str) -> Path; def get_signed_url(self, uri: str, ttl_seconds: int = 3600) -> tuple[str, datetime]`. Local-Backend (filesystem) reicht fuer Phase 2.
    - Tests: `test_orchestrator.py` mit Mock-Pool, submit -> sofortiges queued, get_status nach Pool-Run -> succeeded, summary aus DB. Integration-Test (markiert `@pytest.mark.slow`): echter Pool + echter Popen mit Dummy.otx -> succeeded innerhalb 10 s.
  </behavior>

Damit ist der Pool-Worker engine-frei und der LCG-Singleton im Popen-Subprozess.

Empfehlung: Synchron, separate sync-Engine fuer Callback-Pfad. Begruendung: einfacher, robust, vermeidet Loop-Probleme.
