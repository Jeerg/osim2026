  <behavior>
    - `app/models/model.py` (Erweiterung — die `models`-Tabelle existiert seit Phase 1 mit OTX-Pfad): ergaenze Spalten `format VARCHAR(10) DEFAULT 'otx'`, `payload_json JSONB NULL` (nur fuer format='json'), `schema_hash VARCHAR(64) NULL`. Migration 003 fuegt sie via `op.add_column` hinzu.
    - `schema_service.build_schema()`: ruft `osim_engine.schema.generator.build_json_schema()`, cached via `functools.lru_cache(maxsize=1)`. Cache-Bust via `schema_service.invalidate()` (z.B. nach Engine-Update). Hash via `hashlib.sha256(json.dumps(schema, sort_keys=True).encode()).hexdigest()`.
    - `GET /api/v1/schema/model`: liefert das Schema als `application/schema+json`. Setzt `ETag: "<schema_hash>"`. Bei `If-None-Match: <hash>` -> 304 Not Modified.
    - `GET /api/v1/schema/version`: liefert engine_version + schema_hash + generated_at.
    - `POST /api/v1/models/json`: validiert payload zweistufig:
       1. JSON-Schema-Validation via `jsonschema.Draft202012Validator` (fail-fast: erste 10 Errors als RFC-7807-`errors`-Array).
       2. Engine-Trial-Load: `osim_engine.io.json_loader.load_model_from_json(payload)`; bei Exception 422 mit RFC-7807-`errors`-Array.
       Bei Success: persistiert Row, gibt 201 `ModelJsonResponse` zurueck.
    - `PUT /api/v1/models/{id}/json`: gleiches Validation-Schema; aktualisiert `payload_json`, `updated_at`, `schema_hash`. 404 wenn Modell nicht im Tenant.
    - `GET /api/v1/models/{id}/json`: liefert raw payload-JSON. 404/409 wenn `format != 'json'`.
    - Tenant-Filter analog Phase 1.
    - Tests: `test_schema_endpoint.py` — Schema enthaelt `$defs.PSimulator`, ETag funktioniert, 304 bei If-None-Match. `test_models_json_endpoint.py` — valides Minimal-Modell -> 201, invalides Modell (z.B. unbekannter `$type`) -> 422 mit field-level errors, GET liefert exakt das gespeicherte payload zurueck.
  </behavior>
  <action>
    Engine-Install: pruefe in `pyproject.toml`, dass `osim-engine` als editable installiert ist (Pfad `../engine`). Falls noch nicht: `uv add --editable ../engine`. Ergaenze `jsonschema>=4.21` als Backend-Dependency.

    `schema_service.py`: nutzt `lru_cache` auf einer Modul-Funktion `_build_schema_cached() -> tuple[dict, str, datetime]` (Tuple, damit alle drei Werte gemeinsam invalidierbar sind).

    `model_service.py`: lege Validator-Singleton via `Draft202012Validator(schema)` an (Validator-Instanz cachen, NICHT pro Request neu bauen). Bei Schema-Reload: Validator neu bauen.

    Tests: nutze einen Mini-Fixture-Generator in `tests/conftest.py`: `def minimal_model_json() -> dict` mit 1 PAslEinzel + 1 PDpKnKonstant, referenziert per Knoten-ID-String. Pruefen, dass der Trial-Load durchlaeuft. Bei Schema-Mismatch (z.B. `$type: "unknown.foo"`) muss die Validation 422 zurueckgeben mit JSON-Pointer auf das fehlerhafte Feld.
