(Foundation-Aufruf am Ende des `buildKnotenDetailGrid`.)                                       |

**Wrapper-Indirektion (SCHEMA-MAP.md):** Im OTX-Wire-Format leben die
Assoz-Listen NICHT direkt im PDpKn-Knoten, sondern in separaten
LList-Wrapper-Objekten:

- `knoten.attrs.m_lAssozRess` → OID eines `PAssozRessourceLList`-Wrappers.
  Dessen `sub_refs[0]` enthält die `PAssozBeleg`/`PAssozBetrPers`-OIDs.
- `knoten.attrs.m_lAssozSpeich` → OID eines `PAssozSpeicher`- oder
  `PSpeicherProzLList`-Wrappers (im Real-Modell oft `null`/nicht
  populated). Folge-Lookup für `assoz.attrs.m_lSpeicher` ist analog.
- `assoz.attrs.m_lRessourcen` (für PAssozBeleg) → OID eines
  `PRessBelegLList`-Wrappers. Dessen `sub_refs[0]` enthält die
  `PBetriebsmittel`/`PPerson`-OIDs.

Die TS-Implementation MUSS diesen Wrapper-Pfad genau gehen — direkter
Zugriff auf `assoz.sub_refs[0]` als Fallback ist dokumentiert (SCHEMA-MAP §
„Universelle Regel"), aber sub_refs auf PDpKn-/PAssozBeleg-Ebene ist im
Real-Modell **immer leer**.

## 2. C++-Audit `ConnectKnoten` (Z.338-470)

Setzt die `GLink`-Verbindungen zwischen den im `dFillKnoten` plazierten
Objekten. Jede `GLink`-Konstruktion in C++ trägt einen
`VT_*`-Typ-Discriminator (für Color-Hint + Click-Routing). Tabelle aller
Link-Typen:

| Typ                    | VT_*-Enum-Wert (PGObjBaseViewer.h) | Source-GObject              | Target-GObject              | C++-Zeile      | TS-Konstruktion                                                          |
| ---------------------- | ---------------------------------- | --------------------------- | --------------------------- | -------------- | ------------------------------------------------------------------------ |
| Knoten → AssozRess     | `VT_KNO_ASSRES` (Z.29)             | Knoten-GObj                 | PAssozBeleg / PAssozBetrPers | 372            | `new GLink(knotenGObj, assozGObj)` — Konstruktor registriert in m_OutList/m_InList automatisch. |
| AssozRess → Ressource  | `VT_ASSRES_RESBEL` (Z.31)          | PAssozBeleg-GObj            | PRessBeleg / PBetriebsmittel | 388            | `new GLink(assozGObj, ressGObj)` — eine Linie pro `m_lRessourcen`-Eintrag. |
| AssozBetrPers (Person) | `VT_ASSBETRES_RESBEL` (Z.34)       | PAssozBetrPers-GObj         | PPerson (via `m_lPers`)      | 409            | `new GLink(assozGObj, persGObj)` — Personal-Subgraph.                    |
| AssozBetrPers (Betr.)  | `VT_ASSBETRES_RESBEL` (Z.34)       | PAssozBetrPers-GObj         | PBetriebsmittel (via `m_lBetr`) | 420         | `new GLink(assozGObj, betrGObj)` — Betriebsmittel-Subgraph.              |
| Knoten → AssozSpeich   | `VT_KNO_ASSSPE` (Z.30)             | Knoten-GObj                 | PAssozSpeicher              | 434            | `new GLink(knotenGObj, assozSpeichGObj)`.                                |
| AssozSpeich → ProzSpe. | `VT_ASSSPE_PROSPE` (Z.32)          | PAssozSpeicher-GObj         | PSpeicherProz               | 445            | `new GLink(assozSpeichGObj, speiprozGObj)` — eine Linie pro `m_lSpeicher`-Eintrag. |
| ProzSpe. → Ressource   | `VT_PROSPE_AKT` (Z.33)             | PSpeicherProz-GObj          | PRessBeleg (aktiv)          | 459            | `new GLink(speiprozGObj, ressGObj)` — pro `speiproz.m_lRessourcen`-Eintrag. |

**Spezialfall PAssozBetrPers (Z.394-428):** Pro `betrper`-Eintrag in
`assoz.m_lBetrPers` werden **ZWEI** Links erzeugt — ein primärer zur
Person (`m_lPers`), ein sekundärer zum Betriebsmittel (`m_lBetr`). Im
C++-Code wird zusätzlich ein `GElement` als Tandem-Holder angelegt
(`m_pPrimaryLink` / `m_pSecondaryLink` — siehe `PBetrPersGObj`-Klasse).
In Phase 1.2 reicht **die Doppel-Link-Konstruktion ohne GElement** — das
Tandem-Konzept wird in Phase 4 mit Verschiebungs-Kopplung verdrahtet.

**Phase-1.2-Reduktion:** Die `VT_*`-Typ-Discriminatoren werden NICHT als
Color-Hint visuell unterschieden. Alle Links rendern als generische
GLinks (Default-Farbe `#374151` aus GLink.ts Z.53). Phase 4 könnte
optional die `link.m_crLinkColor` pro VT_*-Typ setzen.

## 3. Listener-Pattern (Z.51-134)

Der C++-`PDlplConnKnotenViewer` hängt einen
`PListenerPDlplConnKnotenViewer`-Subscriber an den `OSimulator`
(`m_listenerSim.Attach(...)`, Z.870). Die Subscriber-Klasse erbt von
`OListenerSimulator` und implementiert vier Event-Handler:

| C++-Listener-Methode | Event-Typ           | C++-Aktion via `OGfxSendMsg`                                                                 | Phase-4-WebSocket-Mapping                                       |
| -------------------- | ------------------- | -------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| `OnPeriodBegin`      | `XM_PERIOD_BEGIN`   | EditorMode OFF, Interaction OFF, `GetGOCtrl()->OnUserMessage(msg, timeCurrent, 0, 0)`        | Subscribe auf `/ws/runs/{run_id}` Event `period_begin{t_begin, t_end}` |
| `OnPeriodEnd`        | `XM_PERIOD_END`     | EditorMode ON, Interaction ON, `GetGOCtrl()->OnUserMessage(...)`                              | WebSocket-Event `period_end{t_end}`                             |
| `OnPeriodBreak`      | `XM_PERIOD_BREAK`   | analog Period_End                                                                            | WebSocket-Event `period_break{t_end}`                           |
| `OnPeriodReset`      | `XM_PERIOD_RESET`   | reset-Trigger im GOCtrl                                                                      | WebSocket-Event `period_reset`                                  |
| `OnSimBegin`         | `XM_SIM_BEGIN`      | EditorMode OFF, Interaction OFF, `GetGOCtrl()->OnUserMessage(...)`                            | WebSocket-Event `sim_begin{t_begin}`                            |
| `OnGfxEvent`         | `XM_GFX_EVENT`      | `GetGOCtrl()->FastDraw(TRUE)` (Live-Repaint-Tick)                                            | WebSocket-Event `gfx_tick{t_current}` (Live-Kennzahl-Update)    |

**Phase-1.2-Vorbereitung:** Der Viewer exportiert einen no-op-Hook
`useSimulationListener({onPeriodBegin, onPeriodEnd, onSimBegin,
onGfxEvent})`. Der Hook hat KEINE Effect-Body, KEINE WebSocket-Logik —
nur die Signatur, damit Phase 4 die Verdrahtung gegen
`/ws/runs/{run_id}` ohne Component-Restruktur einbauen kann. Test 5 der
Spec verifiziert, dass der Hook-Mount **kein** Effect-Triggering
produziert (no-op-Garantie via Threat T-01.2-19).

## 4. TS-Foundation-Reuse

Die Phase-1.1-Foundation deckt alle benötigten Bausteine ab — KEIN neuer
Helper nötig:

| Pattern                                | Foundation-Quelle                                                  | Verwendung                                                                   |
| -------------------------------------- | ------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| Grid-Container                         | `OGraphGrid` (`portal/src/graph/foundation/OGraphGrid.ts`)         | `const grid = new OGraphGrid()` als Root des Detail-Graphs.                  |
| Knoten-Konstruktion                    | `GObjLink` (`portal/src/graph/foundation/GObjLink.ts`)             | `new GObjLink()` für alle Nodes (Knoten, AssozRess, AssozSpeich, Ressourcen, Speicher-Proz). |
| Move/Delete-Forbidden                  | `GObject.SetMoveForbidden`/`SetDeleteForbidden`                    | Für den zentralen Knoten (Z.222-223 des C++).                                |
| Grid-Insertion (mit (col,row)-Coords)  | `OGraphGrid.GOIns(obj, x, y, repaint=false)`                       | Für alle Knoten — `repaint=false`, weil `finalizeLayout` am Ende läuft.      |
| Freie Cell finden                      | `OGraphGrid.GetNextFreeGridPlace({x, y})` (mutiert in-place)       | Für Assoz/Speicher ohne `m_pntRaster` ODER mit belegtem `m_pntRaster`-Ziel.  |
| Link-Konstruktion                      | `GLink(source, target)` (`GLink.ts`)                               | Auto-Register in `m_OutList`/`m_InList` via Constructor.                     |
| Final-Layout (Pixel-Pos berechnen)     | `OGraphGrid.finalizeLayout({x:0, y:0})`                            | Einmal am Ende des `buildKnotenDetailGrid`-Aufrufs.                          |
| Renderer-Konsument                     | `GraphFlowCanvas` (`portal/src/graph/foundation/GraphFlowCanvas.tsx`) | Verlangt `ReactFlowProvider`-Wrapper im Outer-Component (siehe PDurchlaufplanViewerDesign Z.62-73). |
| Dialog-Layout                          | `ChildDialog` (`portal/src/viewers/core/ChildDialog.tsx`)          | Title + Description + Body-Slot.                                             |
| Wire → Grid (Bridge nicht direkt nutzbar) | `wire-to-grid.ts`                                               | Wird hier NICHT verwendet — der Knoten-Detail-Viewer baut ein **eigenes**, schmales Grid, kein Plan-Grid. |

**Was NICHT gebraucht wird:**

- `wireToGrid()` (Plan-Spezifisch — der Detail-Viewer baut ein
  spezielles Grid pro Knoten).
- `GObjSub`/`GLinkSquare`/`GLinkPoint` (alle Links sind generische
  `GLink`, keine Sub-Container, keine rechtwinkligen Routings nötig in
  diesem View).
- Drag-/Connect-Handler (`interactions.ts`) — Viewer ist read-only
  (`disabled ?? true`), keine Mutation.
- `model-store.createObject`/`patchObject`/`appendSubRef` — der Detail-
  Viewer ist eine reine Lesesicht; Mutation kommt in Phase 4 mit dem
  Knoten-Bearbeitungs-Dialog.

## 5. Phase-1.2-Scope-Cuts

Die folgenden C++-Features sind **NICHT** in Welle 1.2-F portiert,
sondern als Phase-4-Backlog dokumentiert:

   `true` (auf dem Knoten-Knoten); auf den Assoz-Objekten könnte man

4. **Context-Menu / OGObjEditGo / Right-Click-Insert:** Phase 4 mit dem
   Mutation-Pfad.
5. **`OnGObjInserted`/`OnGObjRemoved`-Listener:** Mutation-Event-Hooks
   im Grid → Wire-Sync. Phase 4.
6. **`PBetrPersGObj`-Tandem-Element:** Verschiebungs-Kopplung
   Person↔Betriebsmittel. Phase 4.
7. **`VT_*`-Color-Hint:** Visuell unterschiedliche Link-Farben pro
   VT_*-Typ. Backlog Welle 1.3 wenn Bedarf.

## 6. Wire-Schema-Verifikation

Die Phase-1.2-Implementation MUSS gegen folgende Wire-Schemas testen:

```yaml
# PDpKnKonstant (oder Subklasse PDpKn*)
{
  oid: 541,
  klass: "PDpKnKonstant",
  attrs: {
    m_sName: "Knoten A",
    m_lAssozRess: 544,       # OID eines PAssozRessourceLList-Wrappers
    m_lAssozSpeich: null,    # ODER OID eines PAssozSpeicherLList-Wrappers
    m_pntRaster: [2, 0]      # (im Detail-Viewer irrelevant — wird auf (3,0) gemappt)
  },
  sub_refs: [[]]             # immer leer im OTX-Real-Modell
}

# PAssozRessourceLList (Wrapper)
{
  oid: 544,
  klass: "PAssozRessourceLList",
  attrs: {},
  sub_refs: [[421, 426]]     # OIDs der PAssozBeleg-Objekte
}

# PAssozBeleg
{
  oid: 421,
  klass: "PAssozBeleg",
  attrs: {
    m_lRessourcen: 424,      # OID eines PRessBelegLList-Wrappers
    m_pntRaster: [4, 0],     # Positions-Hint, sonst GetNextFreeGridPlace
    m_eStatus: 1
  },
  sub_refs: [[]]
}

# PRessBelegLList (Wrapper)
{
  oid: 424,
  klass: "PRessBelegLList",
  attrs: {},
  sub_refs: [[1137, 1146]]   # OIDs der PBetriebsmittel/PPerson
}
```

Diese Wrapper-Pfad-Verifikation ist die Quelle der Wahrheit; die
Phase-1.2-Implementation iteriert NICHT direkt `assoz.sub_refs[0]`,
sondern liest IMMER `attrs.m_lRessourcen` → Wrapper-Lookup → dessen
`sub_refs[0]`. Identisch für PAssozBetrPers (`m_lBetrPers` → Wrapper
`PBetrPersLList`-ähnlich; im Sample-Modell nicht populated, Pattern aber
analog).
