## 2. Reverse-Index `GetListeKonnektierterKnoten`

**C++ Aufruf (Z.345):**

```cpp
OViewGetRessource()->GetListeKonnektierterKnoten(&m_lKnotenList);
```

Im C++-Original ist `GetListeKonnektierterKnoten` eine **Engine-Side-
Methode** auf der `PRessource`-Basisklasse (PBetriebsmittel/PPerson/
PRessBeleg). Sie sucht über die Engine-Tabellen, welche `PDlplKnoten`
diese Ressource via `m_lAssozRess`-Assoziationen referenzieren.

**In osim-ui (Phase 1.2):** Engine-Reflection ist kein Phase-1.2-Scope.
Wir bauen den Reverse-Index **frontend-side** durch Iteration über
`allObjects`:

```typescript
function buildConnectedKnoten(
  ressOid: number,
  allObjects: Record<number, OBaseObj>,
): OBaseObj[] {
  const result: OBaseObj[] = [];
  for (const o of Object.values(allObjects)) {
    // Nur PDpKn*-Subklassen prüfen
    if (!o.klass.startsWith("PDpKn")) continue;
    // Pro Knoten: Liste der Assoz-Ressourcen via Wrapper-Pfad
    const assozOids = readListViaWrapper(o, "m_lAssozRess", allObjects);
    const hasRef = assozOids.some((aOid) => {
      const assoz = allObjects[aOid];
      if (!assoz) return false;
      // PAssozBeleg: m_lRessourcen via Wrapper-Pfad
      const ressOids = readListViaWrapper(assoz, "m_lRessourcen", allObjects);
      return ressOids.includes(ressOid);
    });
    if (hasRef) result.push(o);
  }
  // Deterministische Sortierung nach oid für stabile Reihenfolge
  result.sort((a, b) => a.oid - b.oid);
  return result;
}
```

**Algorithmus-Komplexität:** `O(K × A × R)` — Anzahl Knoten × Assoz pro
Knoten × Ressourcen pro Assoz. Bei realistischen Phase-1.2-Modellen
(<500 Verknüpfungen, siehe RESEARCH §8.1 R5) bleibt die Build-Zeit unter
der React-16-ms-Frame-Budget. Beim Bosch-Modell (30k+ Objekte) wäre ein
Stress-Test fällig — Backlog Phase 4 oder eigene Performance-Welle.

**Wichtige Wrapper-Indirektion (SCHEMA-MAP.md):** Die `m_lAssozRess`-
und `m_lRessourcen`-Attribute sind **Scalar-OID-Pointer** auf separate
`PAssozRessourceLList`-/`PRessBelegLList`-Wrapper-Objekte. Der
Reverse-Index iteriert daher über drei Indirektions-Stufen:

1. `PDpKn*.attrs.m_lAssozRess` → `PAssozRessourceLList.sub_refs[0]` →
   `PAssozBeleg`-OIDs.
2. `PAssozBeleg.attrs.m_lRessourcen` → `PRessBelegLList.sub_refs[0]` →
   `PBetriebsmittel`/`PPerson`-OIDs.
3. Vergleich mit `ressOid`.

Das `readListViaWrapper`-Helper aus Welle 1.2-F kapselt die Indirektion.

**Source:** `PGObjBaseViewer.h` Z.157-160.


**Phase-4-Aktivierung:** Während eines Sim-Laufs setzt der
`OListenerSimulator` auf `XM_GFX_EVENT` den `m_drawKennzahl`-Flag und
ruft `DrawKennzahl(pDC)` — dieses zeichnet entweder einen vertikalen
Bar pro Ressource (proportional zu `m_dMax`) oder einen Numeric-Overlay
mit dem aktuellen Kennzahl-Wert.

**Phase-1.2-Umsetzung (visueller Placeholder):**

- Eine kleine Box oben rechts im Graph-Pane (`absolute right-2 top-2`).
- Visuell **deaktiviert** dargestellt (`opacity-60`).
- Cyan-Surface-Background (`bg-[var(--color-surface-soft-cyan)]`) +
  Cyan-Border (`border-primary/30`) für 3FLS-EAM-Konformität.
- Label "Kennzahl" + Uppercase-Subtext "Phase 4" für visuelle
  Disambiguation gegen aktive Kennzahlen.
- ARIA-Label und `title`-Attribut erklären, dass dies Phase 4
  aktiviert.
- `data-slot="kennzahl-placeholder"` für E2E + Test-Selektoren.

**KEIN Backend-Call** in Phase 1.2 — der Slot ist rein visuell. Phase 4
ersetzt den Placeholder durch eine aktive `<KennzahlBarDisplay>`-
Component, die `m_iKnz`-Werte aus dem WebSocket-Tick konsumiert.

**Wichtig zu wissen:** `m_drawKennzahl` ist im C++-Code an die
`m_listenerSim`-Attach-Logik gekoppelt (siehe Welle 1.2-F-Audit §3) —
ohne aktiven Sim-Run bleibt der Slot inaktiv. Phase 1.2 imitiert das
mit dem `disabled`-Look ohne den Listener tatsächlich zu mounten.

## 4. TS-Foundation-Reuse

Die Phase-1.1-Foundation deckt alle Bausteine ab — der Viewer ist
strukturell eine **Variante** des `PDlplConnKnotenViewer` (Welle 1.2-F)
mit drei Unterschieden:

| Aspekt                       | Welle 1.2-F (Knoten-zentriert)                       | Welle 1.2-G (Ressource-zentriert)                                                       |
| ---------------------------- | ---------------------------------------------------- | --------------------------------------------------------------------------------------- |
| Zentrales GObj               | `PDlplKnoten` bei `(3, 0)`                            | **Ressource** (PBetriebsmittel/PPerson/PRessBeleg) bei `(knotenCount/2, 0)`             |
| Iterations-Quelle            | `knoten.m_lAssozRess` (direkter Wrapper-Pfad)        | **Reverse-Index** über alle `PDpKn*`-Objekte                                            |
| Zusatz-UI                    | nur Graph-Pane                                       | **TTY-Pane links + Kennzahl-Slot-Placeholder rechts oben**                              |
| Speicher-Assoz-Pfad          | voll portiert (PAssozSpeicher + PSpeicherProzLList)  | NICHT portiert (Backlog Welle 1.3)                                                      |
| Listener-Hook                | `useSimulationListener` exportiert + mountet         | identisch — **re-use** aus 1.2-F (kein zweiter Pfad)                                    |

**Re-use aus Welle 1.2-F (1:1):**

- `readListViaWrapper(parent, attrName, allObjects)` — Wrapper-Pfad-
  Resolver.
- `readListDirect(parent, attrName)` — Fallback für aufgelöste Arrays.
- `readRaster(attrs)` — `m_pntRaster`-Lesehilfe.
- `labelFor(obj)` — Label-Konvention.
- `placeWithFallback(grid, gObj, raster)` — Cell-Plazierungs-Hilfe (in
  1.2-G aber selten, weil Layout flach und linear).
- `useSimulationListener(handlers)` — Phase-4-Vorbereitung als no-op-
  Hook.

**Importpfad:** Die Helper-Functions bleiben in
`portal/src/viewers/PDlplConnKnoten/PDlplConnKnotenViewer.tsx` und
werden NICHT in ein gemeinsames Modul geliftet (vorerst). Nur
`useSimulationListener` wird als named export bereits konsumiert.
Re-Refactor auf ein gemeinsames `viewers/PDlpl/utils.ts`-Modul ist
Backlog Welle 1.3, falls ein dritter Konsument auftaucht (Threat
T-01.2-22 dokumentiert das als akzeptiertes Boundary-Cross).

**Neu in Welle 1.2-G:**

- `buildConnectedKnoten(ressOid, allObjects)` — Reverse-Index-Builder.
- `buildRessVerknuepfungGrid(ressObj, connectedKnoten, allObjects)` —
  baut das Grid mit Ressource zentral + Knoten in Reihe 0 + Assoz-Nodes
  in Reihe 1 + 2 Links pro Assoz.
- `KennzahlSlotPlaceholder` — visueller Phase-4-Slot.
