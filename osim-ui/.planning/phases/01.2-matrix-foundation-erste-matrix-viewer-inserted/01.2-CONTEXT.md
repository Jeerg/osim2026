<decisions>
## Implementation Decisions (locked)

### Architektur

1. **Cells sind Wire-Daten-getrieben, NICHT GObject-basiert.**
   Im OSim2004-Original sind Matrix-Cells `GObject`-Instanzen (z.B. `PRessBelegMatrixGObj`). In osim-ui mappen wir Cells direkt auf Wire-Records (`{ rowOid, colOid, value, status }`). Vorteil: einfache Persistierung über bestehenden `model-store`-Pfad; Nachteil: keine Cell-eigene Grid-Position-Mutation. Da Cells in einer Matrix-Tabelle leben (nicht in einem freien Grid), brauchen sie keine `m_pntRaster`-Logik.

2. **DrawCells-Hook ist ein React-Component-Slot, nicht ein Canvas-Callback.**
   Foundation liefert `<MatrixGrid>` mit `renderCell={(row, col, value) => ReactNode}`-Prop. Konsumenten geben Custom-Components. Im Gegensatz zum C++ (`CDC*`-Drawing) bleibt es Browser-/React-nativ.

3. **Header sind sticky via CSS `position: sticky`.**
   Kein eigener Sticky-Mechanismus in der Foundation — Standard-Web-Pattern, funktioniert mit `overflow: auto` auf Container + `position: sticky; top: 0` auf Header-Row.

4. **Inline-Edit-State-Machine ist Foundation-eigene Hook.**
   `useInlineCellEdit({ cellId, value, onCommit, readOnly })` returnt `{ editing, startEdit, commit, cancel }`. Foundation-agnostisch — kann auch außerhalb Matrix-Viewern genutzt werden.

5. **Block-Select-Selection ist State im Konsumenten (model-store-Slice oder lokaler React-State), Foundation liefert nur Helper-Funktionen.**
   `computeCellRange(start, end)`, `isInRange(cell, range)`, `extendSelection(range, cell)` — pure-functions.

6. **Clipboard-Format ist JSON in `application/x-osim-matrix-cells`-MIME-Type.**
   Cross-Viewer-Paste über Browser-DataTransfer. Standardisiert: `{ origin: <viewerKlass>, cells: [{ row, col, value }] }`.

### Pattern: ViewerProps für Matrix-Viewer

Bestehende `ViewerProps`-Convention (aus PDurchlaufplanViewerDesign):
```ts
type ViewerProps = {
  obj: OBaseObj;       // das gerade angezeigte Wire-Objekt
  allObjects: ...;     // Index aller geladenen Wire-Objekte
  schema: ClassSchema;
  disabled?: boolean;
}
```

Matrix-Viewer brauchen evtl. `relatedObjects`-Hint (welche Wire-OIDs als Zeilen/Spalten). Foundation-Hook könnte sein: `useMatrixData({ rowOids, colOids, cellLookup, edit })`.

### Pattern: Multi-Source-Audit aus Phase 1.1

Welle G1–G26 hat gezeigt: parallel-Lese-Pässe (C++-Audit + TS-Audit) sind das Standard-Pattern für 1:1-Reproduktion. Planner sollte pro Viewer-Welle einen C++-Audit-Step einplanen, BEVOR die TS-Implementation losgeht.

### Listener-Framework-Vorgriff

PRessVerknuepfungViewer hat im Original Kennzahl-Anzeige während Sim-Lauf (`m_drawKennzahl` Flag). In Phase 1.2 reicht ein **Placeholder-Slot** in der UI (Cyan-Disabled-Box mit "Phase 4"-Hinweis), damit Phase 4 nur den Hook einbauen muss, nicht die Component restrukturieren.

</specifics>
