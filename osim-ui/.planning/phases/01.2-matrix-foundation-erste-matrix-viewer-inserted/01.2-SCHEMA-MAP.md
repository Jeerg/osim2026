## Kernerkenntnis (gegen die Plan-Annahme prüfen!)

Die im Plan unter <action> formulierte Frage *„welcher sub_refs[N]-Slot trägt
`PAssozBeleg.m_lRessourcen`?"* basiert auf einer **falschen Annahme**. Die
Sondierung hat ergeben:

> **Weder `PAssozBeleg` noch `PDpKn*` halten ihre Listen direkt in
> `sub_refs[N]`.** Beide haben nur einen leeren `sub_refs[0]`-Slot. Die
> Listen leben in **separaten LList-Wrapper-Objekten**, deren OIDs als
> Scalar-Pointer in `attrs.m_l*` liegen.

Konkret im Sample-Modell (PAssozBeleg oid=421):

```
PAssozBeleg oid=421
  attrs.m_lRessourcen      = 424       (scalar OID pointer)
  attrs.m_lKnoten          = None      (in diesem Modell ungenutzt)
  attrs.m_LinkStatusList   = 425
  attrs.m_lViewInf         = 422
  attrs.m_lUnterAssoz      = 423
  sub_refs[0]              = []        (LEER!)
  inline_children          = 4

PRessBelegLList oid=424
  sub_refs[0] = [1137(PBetriebsmittel), 1146(PBetriebsmittel), 1158(PBetriebsmittel)]
```

Und symmetrisch für die Knoten-Seite (PDpKnKonstant oid=541):

```
PDpKnKonstant oid=541
  attrs.m_lAssozRess  = 544          (scalar OID pointer)
  attrs.m_lAssozSpeich = None
  attrs.m_lProzesse   = 543
  sub_refs[0]         = []           (LEER!)

PAssozRessourceLList oid=544
  sub_refs[0] = []                   (in diesem Knoten leer, kein Assoz)
```

Und auf der „echten" Belegungs-Seite — `PDpKnMengeRuesten` mit gefülltem
Wrapper (oid=417):

```
PDpKnMengeRuesten oid=417
  attrs.m_lAssozRess  = 420          (scalar OID pointer)
  sub_refs[0]         = []

PAssozRessourceLList oid=420
  sub_refs[0] = [421(PAssozBeleg), 426(PAssozBeleg), 431(PAssozBeleg), 437(PAssozBeleg)]
```

**Konsequenz:** Plan 06 Task 2 muss `_slotToAttrName` **nicht** für
`PAssozBeleg` oder `PDpKn*` erweitern — diese Klassen brauchen kein
Slot→Attr-Mapping. Die Erweiterung muss stattdessen die **Wrapper-Klassen**
abdecken (`PRessBelegLList`, `PAssozRessourceLList`, `PDlplKnotenLList`,
…). Außerdem braucht der Cell-Create-Pfad zwei zusätzliche Schritte:

1. Vor dem appendSubRef sicherstellen, dass die LList-Wrapper-Instanz
   existiert. Wenn `parent.attrs.m_lRessourcen === null`, muss eine neue
   `PRessBelegLList` per `createObject` angelegt und ihre OID in
   `parent.attrs.m_lRessourcen` geschrieben werden.
2. Der `appendSubRef`-Call zielt dann auf den Wrapper (`wrapperOid, 0, …`),
   nicht auf den PAssozBeleg/PDpKn-Parent.

### PDlplKnoten (Basisklasse — im Sample-Modell **nicht** direkt instanziiert)

würde, sollte sie das gleiche `attrs.m_lAssozRess`-Pointer-Schema verwenden
wie die Subklassen (Polymorphismus in der Engine-Wire-Definition).

## Konsequenzen für Plan 06 Task 2 (`_slotToAttrName`)

Aktuelle Implementation in `portal/src/stores/model-store.ts` Z.133–139:

```typescript
function _slotToAttrName(klass: string, slot: number): string | null {
  if (klass === "PDurchlaufplan") {
    if (slot === 0) return "m_lKnoten";
    if (slot === 1) return "m_lKanten";
  }
  return null;
}
```

`PDurchlaufplan` hat im Plan-Alias-Schema 2 sub_refs-Slots (`m_lKnoten`,
`m_lKanten`). Das ist eine UI-/Convenience-Annahme aus Welle 1.1-G20-C und
unterscheidet sich vom echten OTX-Wrapper-Pattern (`PDlplKnotenLList`
+ `PDlplKanteLList`) — das ist **konsistent**, weil der UI-Wire-Stream
auf flache Container-Klassen optimiert wurde und Phase 4's
Engine-Reflection das später nachzieht.

**Patch-Vorschlag für die LList-Wrapper-Familie:**

```typescript
function _slotToAttrName(klass: string, slot: number): string | null {
  if (klass === "PDurchlaufplan") {
    if (slot === 0) return "m_lKnoten";
    if (slot === 1) return "m_lKanten";
  }
  // LList-Wrapper-Familie: jeder Wrapper hat genau einen sub_refs[0]-
  // Slot. Der attrName ist NICHT generisch ableitbar; jeder Wrapper-Typ
  // hat einen anderen Eltern-attrName, in den die Wrapper-OID geschrieben
  // wird. Das Mapping passiert beim Cell-Create-Pfad in zwei Schritten:
  //
  //   1. Wrapper-OID anlegen (createObject), Eltern-attrs.m_l*-Pointer
  //      setzen (patchObject).
  //   2. appendSubRef(wrapperOid, 0, childOid) — kein attrName-Sync nötig,
  //      weil der Wrapper selbst kein m_l*-Attribut hat (sub_refs[0] ist
  //      die einzige Daten-Quelle).
  //
  // Daher: KEIN Mapping-Eintrag für PRessBelegLList / PAssozRessourceLList
  // / etc. — der Cell-Create-Pfad managed das selbst.
  return null;
}
```

**Wichtig — was Plan 06 Task 2 NICHT tun darf:** keine Einträge der Form
`if (klass === "PAssozBeleg" && slot === 0) return "m_lRessourcen"` —
diese würden sub_refs[0] (das immer leer ist) fälschlich mit
attrs.m_lRessourcen (das ein scalar Pointer ist, KEIN Array) synchronisieren
und korrumpieren das Schema.

**Wichtig — was Plan 06 Task 2 stattdessen tun muss:**

```typescript
// In PRessBelegMatrixViewer (oder dedizierter Cell-Logik):
function ensurePRessBelegWrapper(assozBelegOid: number): number {
  const parent = useModelStore.getState().wire.objects[assozBelegOid];
  let wrapperOid = parent.attrs.m_lRessourcen as number | null;
  if (wrapperOid === null) {
    wrapperOid = useModelStore.getState().createObject("PRessBelegLList", {});
    useModelStore.getState().patchObject(assozBelegOid, { m_lRessourcen: wrapperOid });
  }
  return wrapperOid;
}

// Cell-create:
const wrapperOid = ensurePRessBelegWrapper(assozBelegOid);
useModelStore.getState().appendSubRef(wrapperOid, 0, childBetriebsmittelOid);
```

Analog für `m_lAssozRess` (Wrapper-Klasse `PAssozRessourceLList`,
Eltern-Klassen alle `PDpKn*`).

## Test-Strategie für Plan 06

Damit der Wrapper-Indirektions-Pfad nicht erneut Korruption produziert (vgl.
PRessVerknuepfung-Stub-Erfahrung), MUSS Plan 06's Cell-Create-Spec
folgende Assertions enthalten:

1. **Wrapper-Lazy-Create:** Wenn `parent.attrs.m_lRessourcen === null`, dann
   nach Cell-Edit:
   - Ein neues `PRessBelegLList`-Objekt existiert.
   - `parent.attrs.m_lRessourcen === wrapperOid` (NICHT `null`).
   - `wire.objects[wrapperOid].sub_refs[0]` enthält die OID des neuen
     PBetriebsmittel/PAssozBeleg.

2. **Wrapper-Reuse:** Wenn `parent.attrs.m_lRessourcen` bereits eine OID
   trägt, dann nach Cell-Edit:
   - KEIN neues `PRessBelegLList`-Objekt entstanden.
   - `parent.attrs.m_lRessourcen` unverändert.
   - `wire.objects[wrapperOid].sub_refs[0]` um eine OID länger.

3. **Cell-Delete:** Nach Cell-Wert = 0/leer:
   - Die Ziel-OID ist aus `wire.objects[wrapperOid].sub_refs[0]` entfernt.
   - `parent.attrs.m_lRessourcen` unverändert (Wrapper bleibt bestehen,
     auch wenn `sub_refs[0]` jetzt leer ist).

4. **Persistenz-Round-Trip:** Nach Save+Reload sind alle drei Properties
   (parent.attrs.m_lRessourcen, wrapperOid, wrapperOid.sub_refs[0]) bit-
   identisch zum Vor-Save-Zustand. Diese Assertion fängt Save-Pfad-Bugs
   à la Welle G16.
