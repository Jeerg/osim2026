# Phase 01.2 Plan 05: Matrix-Foundation Welle D Summary

Foundation-Welle 1.2-D der Matrix-Foundation: Clipboard-Format und
DataTransfer-IO für Cross-Viewer-Block-Copy/Paste (CONTEXT D-6).
Standardisiertes JSON-Payload in einem Custom-MIME-Type
`application/x-osim-matrix-cells` — Pattern 1:1 aus
`PlanToolbar.tsx` (Welle G17-A) übernommen, das für Knoten/Kanten-Drag
denselben Mechanismus mit `application/x-osim-plan-toolbar` nutzt.
Cell-`row`/`col` sind RELATIVE Offsets zum Block-Ursprung (analog
C++ `PRessBelegMatrix_CellCopyStruct.relpoint`, gefüllt via
`m_pRPos = absolutePoint - originPoint` in
`PRessBelegMatrixViewer.cpp` Z.1079). Konsument (Welle 1.2-06
PRessBelegMatrixViewer) rechnet beim Paste:
`targetCell = pasteOrigin + payload.cells[i]`.

Drei explizite STRIDE-Mitigations sind im Code verankert und über Specs
verifiziert: (T-01.2-10) **KEIN** `text/plain`-Fallback in
`writeToDataTransfer` — Custom-MIME isoliert den Payload von externen
Apps; Test 7 prüft dass `dt.listTypes()` ausschließlich
`MATRIX_CLIPBOARD_MIME` enthält. (T-01.2-12) `deserializeClipboard` ist
defensive Pipeline mit 6 Form-Branches (Input-typeof, JSON.parse
try/catch, null-Object-Check, origin-string-Check, cells-array-Check,
Cell-row/col-number-Check) — jeder Branch returnt null statt zu werfen;
Tests 4–6b decken alle ab. (T-01.2-11) `checkOriginCompatible` ist
Pure-Function default-deny mit optionaler `compatMap` — Konsumenten
entscheiden vor Paste, ob ein Cross-Viewer-Paste erlaubt ist.

## Known Stubs

Keine — alle 6 Exports sind voll funktionsfähig. Was bewusst NICHT im
Modul ist (per CONTEXT D-6 / Plan-Boundary):

- **KEIN `navigator.clipboard.read/write`-Wrapper** — DataTransfer
  reicht für Copy/Paste-via-Browser-Events (DragStart/Drop und
  ClipboardEvent.clipboardData bei Cut/Copy/Paste). Globaler Clipboard-
  Read in einem DOM ohne aktive Drag/Copy-Event-Context ist
  Phase-1.3-Backlog (würde Permission-Prompts triggern und ist nicht
  schmerzfrei testbar).
- **KEIN `text/plain`-Fallback** — bewusste T-01.2-10 Mitigation, siehe
  Test 7 + Decision oben.
- **KEINE Default-Compat-Map** — Foundation kennt keine konkreten
  Viewer-Klassen. Konsumenten (Welle 06+ PRessBelegMatrixViewer)
  liefern ihre eigene compatMap aus ihrem Domänen-Wissen.
- **KEINE Cell-`value`-Validierung** — Foundation kennt das Wire-Schema
  nicht. `deserializeClipboard` validiert nur Struktur (row, col); der
  Konsument muss `value` vor dem Paste-Apply gegen sein PropertySchema
  prüfen (T-01.2-07-Analog: Type-Parse-Boundary in der Foundation,
  Wert-Validierung beim Konsumenten).
- **KEINE Block-Range-Inference** — der Konsument iteriert die
  `useBlockSelection`-Range (Welle C) und konstruiert das `cells`-Array
  selbst, indem er pro Cell-Offset value liest. Foundation
  serialisiert/deserialisiert nur den fertigen Payload.
