# Phase 1.2: Matrix-Foundation + erste Matrix-Viewer — Research

**Researched:** 2026-05-25
**Domain:** Frontend-Foundation-Erweiterung (Matrix-Cell-Rendering + 2 Graph-Detail-Viewer) auf bestehender GraphObject-Foundation (Welle G1–G27)
**Confidence:** HIGH (C++-Original direkt gelesen, TS-Foundation direkt geprüft, Bestands-Stubs der drei Ziel-Viewer identifiziert und analysiert)

## Summary

Phase 1.2 sieht auf den ersten Blick wie ein Foundation-Ausbau aus (Matrix-Cell-Rendering, Sticky-Headers, Inline-Edit, Block-Select/Copy/Paste). Tatsächlich ist die wichtigste Erkenntnis dieser Recherche: **In Phase 1 wurden bereits Stubs für alle drei Ziel-Viewer angelegt** (`portal/src/viewers/PRess/PRessBelegMatrixViewer.tsx`, `PRessVerknuepfungViewer.tsx`, sowie ein generischer `matrix-common.tsx` mit shadcn-Table-Pattern). Diese Stubs sind **konzeptionell falsch** für die OSim2004-Treue — der bestehende `PRessBelegMatrixViewer` zeigt eine flache PRessBeleg-Property-Liste (Spalten = Properties: Name/Kapazität/Kosten), das C++-Original ist aber eine echte 2D-Matrix mit **Zeilen = PRessBeleg, Spalten = PDlplKnoten, Cells = PAssozBeleg-Assoziationen**. Phase 1.2 muss die Stubs daher ersetzen/refactorieren, nicht nur erweitern.

Foundation-seitig ist die bisherige GraphObject-Pipeline (`OGraphGrid`, `wire-to-grid`, `view-adapter`) für freie 2D-Graphen ausgelegt, nicht für Tabellen-Matrizen mit synchronisierten Headers. Die im CONTEXT festgeschriebene Entscheidung D-2 (DrawCells-Hook als React-Component-Slot, NICHT als Canvas-Callback) sowie D-1 (Cells sind Wire-Daten-getrieben, keine GObject-Instanzen) folgen genau dieser Trennung: die neue `matrix/`-Foundation läuft **parallel** zur Graph-Foundation, nutzt aber dieselben Tokens, denselben Wire-/model-store-Pfad und dieselben Pattern-Konventionen (ViewerProps, ChildDialog, data-attribute-Selektoren).

Die beiden Graph-Detail-Viewer (PDlplConnKnotenViewer, PRessVerknuepfungViewer) sind eigentlich **kein neues Pattern** — sie sind GraphObject-Konsumenten wie PDurchlaufplanViewerDesign (Phase 1.1), nur mit anderem Wire-Knoten-Typ (PAssozBeleg, PAssozSpeicher, PAssozBetrPers) und anderen Link-Typen (`PAssozGLink` mit Status statt PDlplKante). Sie können die bestehende Foundation direkt nutzen.

**Primary recommendation:** Drei strikt getrennte Sub-Phasen (Welle 1.2-A bis -D Foundation; Welle 1.2-E Matrix-Viewer-Ersatz; Welle 1.2-F+G Graph-Detail-Viewer; Welle 1.2-H E2E). Die bestehenden Stubs müssen FRÜH (vor Welle E) als bewusste Lösch-/Refactor-Entscheidung sichtbar gemacht werden, damit der Planner keine versehentlichen Add-On-Wellen plant, die mit dem Original konkurrieren.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Architektur**

1. **Cells sind Wire-Daten-getrieben, NICHT GObject-basiert.** Im OSim2004-Original sind Matrix-Cells `GObject`-Instanzen (z.B. `PRessBelegMatrixGObj`). In osim-ui mappen wir Cells direkt auf Wire-Records (`{ rowOid, colOid, value, status }`). Vorteil: einfache Persistierung über bestehenden `model-store`-Pfad; Nachteil: keine Cell-eigene Grid-Position-Mutation. Da Cells in einer Matrix-Tabelle leben (nicht in einem freien Grid), brauchen sie keine `m_pntRaster`-Logik.

2. **DrawCells-Hook ist ein React-Component-Slot, nicht ein Canvas-Callback.** Foundation liefert `<MatrixGrid>` mit `renderCell={(row, col, value) => ReactNode}`-Prop. Konsumenten geben Custom-Components. Im Gegensatz zum C++ (`CDC*`-Drawing) bleibt es Browser-/React-nativ.

3. **Header sind sticky via CSS `position: sticky`.** Kein eigener Sticky-Mechanismus in der Foundation — Standard-Web-Pattern, funktioniert mit `overflow: auto` auf Container + `position: sticky; top: 0` auf Header-Row.

4. **Inline-Edit-State-Machine ist Foundation-eigene Hook.** `useInlineCellEdit({ cellId, value, onCommit, readOnly })` returnt `{ editing, startEdit, commit, cancel }`. Foundation-agnostisch — kann auch außerhalb Matrix-Viewern genutzt werden.

5. **Block-Select-Selection ist State im Konsumenten** (model-store-Slice oder lokaler React-State), Foundation liefert nur Helper-Funktionen. `computeCellRange(start, end)`, `isInRange(cell, range)`, `extendSelection(range, cell)` — pure-functions.

6. **Clipboard-Format ist JSON in `application/x-osim-matrix-cells`-MIME-Type.** Cross-Viewer-Paste über Browser-DataTransfer. Standardisiert: `{ origin: <viewerKlass>, cells: [{ row, col, value }] }`.

**Verzeichnis-Struktur**

- Foundation-Code in `portal/src/graph/foundation/matrix/` (neuer Unterordner, separiert von Graph-Foundation)

## 1. C++-Original-Befunde pro Ziel-Viewer

### 1.1 `PRessBelegMatrixViewer` (kanonischer Matrix-Viewer)

**Klassen-Hierarchie (C++):**

```
PRessBelegMatrixViewer       : OViewerChildDialog     (Top-Level Container)
├── m_lKnoten                : CList<oprPDlplKnoten>  (Spalten-Quelle)
├── m_lRessBeleg             : CList<oprPRessBeleg>   (Zeilen-Quelle)
├── m_Tree                   : PRessBelegMatrixTreeCtrl  (Sidebar-Filter, Phase 1.2 deferred)
├── m_cbView                 : CComboBox  (View-Modi: ALL/PERS/RESS)
├── m_cbVerkStatus           : CComboBox  (Status: PREFER/STD/IF_NEEDED/BLOCKED)
└── GetGOCtrl()              : PRessBelegMatrixOGCtrl  (Canvas + Mechanik)
     └── STDGrid             : PRessBelegMatrixGrid : OGraphGrid  (2D-Raster)
          └── DrawCells(dc)  : zeichnet Pen-Shadow + Pen-Height pro Cell-Feld
```

**Wichtigste Methoden:**

- **`dframeFill()`** (cpp Z.1476): Top-Level-Pipeline — `dframeClear()` → `dFillKnotenList()` → `dFillRessList()` → `GetSTDGrid()->SetGridSize(CSize(knoten,res))` → `RefillTree()`. Re-Build der gesamten Matrix bei jeder Mutation.
- **`OnLButtonDown`** (cpp Z.825): Click auf leere Cell → `GetRessAtRaster(rasterpoint)` + `GetKnotenAtRaster(rasterpoint)` → `GetAssozBeleg(knoten,beleg)` (entweder neue PAssozBeleg oder beste vorhandene) → `assress->m_lRessourcen->AddTail(beleg)` → optional `SetBaseLinkStatus` mit Combobox-Wert → `RebuildMatrix()`.
- **`OnRButtonDown`** (cpp Z.963): Rechtsklick auf BELEGTE Cell → `RemovePSimObj(beleg)` (entfernt Belegung) → ggf. Tree invalidate → `RebuildMatrix()`.
- **`OnLButtonDblClk`** (cpp Z.905): Doppelklick → öffnet `PDpKnBetrPersMatrixViewer` (Sub-Matrix für den Knoten). Phase 1.2: deferred.
- **`CopyCell2Buffer(p, copyblock)`** (cpp Z.1079): Speichert `{ scrCellAssRess, scrCellRessBeleg, scrCellKnoten, relpoint }` als `PRessBelegMatrix_CellCopyStruct` in `m_lCopyBuffer`. Relativ-Position zur Block-Ecke.
- **`PasteCellFromBuffer(org)`** (cpp Z.1116): Iteriert Buffer, ruft `PasteCell(org, cell)` → berechnet `rasterpoint = org + cell.m_pRPos` → wenn Ziel-Cell leer + Ress/Knoten existieren: `GetPasteAssoz(beleg,knoten,cell)` + `assress->m_lRessourcen->AddTail(beleg)`.
- **`OnDrawFrame(grid, dc)`** (cpp Z.406): zeichnet die zwei Header-Zonen (Col-Header oben mit Knoten-Namen + Sub-Plan-Farb-Hintergrund; Row-Header links mit Ressourcen-Namen + Klass) + Corner-Cell mit "DlplKnoten / Bel.ress."-Beschriftung + 1px-Trennlinien (PenHeight + PenShad).
- **`PRessBelegMatrixGrid::DrawCells(dc)`** (cpp Z.163): zeichnet 3D-Cell-Ränder (Shadow + Highlight) pro besetzter Cell — **bereits in Welle G21 in TS portiert** als `GridBackground.tsx`. Wiederverwendbar für Matrix.

**Cell-Visualisierung:** `PRessBelegMatrixGObj::DrawSymbol(dc)` (cpp Z.60) — schreibt Status-Text in die obere Cell-Hälfte (`"bevorzugt" / "standard" / "notfalls" / "geblockt"` je nach `m_eStatus`). DrawText (cpp Z.109) schreibt `m_string` in die untere Hälfte (ist normalerweise der Ressource-Name oder leer).

**Combobox-Modi:**
- `m_cbView` (ALL/PERS/RESS): filtert Zeilen — alle Belegungs-Ressourcen / nur Personen / nur Betriebsmittel
- `PRBM_AND` vs `PRBM_OR` (Verknüpfungs-Modus): bei AND wird IMMER ein neues PAssozBeleg pro Cell erzeugt; bei OR wird ein vorhandenes PAssozBeleg gesucht (`_SeekBestAssoz`)

### 1.2 `PDlplConnKnotenViewer` (Knoten-Detail-Graph)

**Klassen-Hierarchie:**

```
PDlplConnKnotenViewer        : PGObjBaseViewer        (Graph-Viewer-Base)
├── PListenerPDlplConn..     : OListenerSimulator     (Sim-Events, Phase 1.2 deferred)
├── m_MoveList               : CList<GObjMoveHelper*> (Move-Helper für Konsistenz)
├── m_RemoveList             : CList<GConnViewer..*>  (Remove-Helper)
├── m_cbView                 : CComboBox              (View-Modus)
└── GetGOCtrl()              : PGObjViewerCtrl        (Graph-Canvas)
```

**Pipeline:**

- **`dFillKnoten(knoten, grid)`** (cpp Z.149): zentrale Layout-Methode. Pipeline:
  1. Knotenname als Grid-Titel setzen (`grid->SetText`)
  2. Für jede Assoziation in `knoten->m_lAssozRess`: Assoz-GObj erzeugen + an Position `assozres->m_pntRaster` einfügen (`grid->GOPIns`). Wenn Position belegt → in `assGgobjList` für späteren freien Platz.
  3. Speicher-Assoziation `knoten->m_lAssozSpeich` analog einfügen.
  4. **Den Knoten selbst** als `PDlplKnotenGObj` an fester Position `CPoint(3,0)` einfügen, mit `SetMoveForbidden(TRUE)` + `SetDeleteForbidden(TRUE)`.
  5. Übrige Assoz-GObjs an `GetNextFreeGridPlace` einfügen + `pntRaster` zurückschreiben.

- **`ConnectKnoten(knoten)`** (cpp Z.338): erzeugt alle Links:
  - Knoten → AssozRess: `VT_KNO_ASSRES` (für PAssozBeleg + PAssozBetrPers)
  - AssozRess → einzelne Ressourcen (in `m_lRessourcen`): `VT_ASSRES_RESBEL`
  - Knoten → AssozSpeich: `VT_KNO_ASSSPE`
  - AssozSpeich → SpeicherProz: `VT_ASSSPE_PROSPE`
  - SpeicherProz → aktive Ressourcen: `VT_PROSPE_AKT`
  - **Spezialfall PAssozBetrPers:** pro `betrper` zwei Links — einmal zu `m_lPers` und einmal zu `m_lBetr` (Personal-Betriebsmittel-Tandem mit Primary/Secondary)

- **Listener (`PListenerPDlplConnKnotenViewer`):** wird auf Sim-Lauf gehört (XM_PERIOD_BEGIN/END/BREAK/RESET/SIM_BEGIN/GFX_EVENT). **Phase 1.2: komplett deferred** — Hook-Slot ohne Body anlegen, Phase 4 verdrahtet.

**Layout-Konvention:** Knoten zentral bei Grid-Position `(3, 0)`, Assoz-Objekte links/rechts/oben/unten in den Zellen 0..2 + 4+. Speicher-Assoz bekommt eine eigene Reihe via `GetNextFreeGridPlace`.

### 1.3 `PRessVerknuepfungViewer` (Ressource-zu-Knoten-Graph mit Kennzahl-Slot)

**Klassen-Hierarchie:**

```
PRessVerknuepfungViewer      : OViewerChildDialog     (Top-Level)
├── m_lKnotenList            : CList<oprPDlplKnoten>  (alle verknüpften Knoten)
├── TTY()                    : Text-Output-Pane       (links: Ress-Name + Klassname)
└── GetGOCtrl()              : PGObjViewerCtrl        (rechts: Graph)
     └── 2 STDGrids:
          - Grid 0: leer (Ressource oben links)
          - Grid 1: dFillResourcenGitter (Ress-Gitter)
```

**Pipeline:**

- **`dframeFill()`** (cpp Z.337): `OViewGetRessource()->GetListeKonnektierterKnoten(&m_lKnotenList)` → Engine liefert alle Knoten, die diese Ressource referenzieren → `dFillResourcenGitter(GetGOCtrl()->GetAtIndex(1))` → `dFillKnoten()`.

Vorher wird `dFillAssoz(knoten, laufx, gknoten)` aufgerufen — diese sammelt für PRessMenge: `PAssozMengeAbfr/Verbr/Erzgt`-GObjs; für PRessBeleg: `PAssozBeleg`-GObj + ggf. `PAssozSpeicher`-GObj wenn der Knoten einen aktiven Speicher hat, der die Ress referenziert.

### 4.2 Matrix-Cell-Lookup-Logik (1:1 zum C++)

**Read-Pfad (rendern):**

```typescript
// für Cell (rowRessOid, colKnotenOid):
const knoten = allObjects[colKnotenOid];
const assozRessOids = knoten.attrs.m_lAssozRess as number[] | undefined
  ?? knoten.sub_refs[<slot-for-assoz>] ?? [];
const assoz = assozRessOids
  .map(oid => allObjects[oid])
  .find(a => a?.klass === "PAssozBeleg"
    && (a.attrs.m_lRessourcen as number[] | undefined)?.includes(rowRessOid));
// assoz === undefined  → Cell ist leer
// assoz !== undefined  → Cell hat Wert; m_eStatus zeigt PREFER/STD/IF_NEEDED/BLOCKED
```

**Write-Pfad (Cell-Edit / Cell-Create):**

```typescript
// Leere Cell + User klickt + wählt Status:
// 1. Existiert schon ein PAssozBeleg im Knoten? (View-Modus PRBM_OR = Best-Match-Suche; PRBM_AND = immer neu)
//    PRBM_OR: SeekBestAssoz(knoten, beleg) — sucht erstbeste PAssozBeleg mit kompatiblem Typ
//    PRBM_AND: immer new PAssozBeleg
// 2. PAssozBeleg.m_lRessourcen.push(rowRessOid)  →  appendSubRef(assoz.oid, slot, rowRessOid)
// 3. Optional PAssozBeleg.m_eStatus = userStatus  →  patchObject(assoz.oid, { m_eStatus: ... })
```

### 5.1 Bibliotheks-Entscheidungen

**Keine neuen Bibliotheken nötig.** Alle Foundation-Komponenten sind aus dem Bestand baubar:

| Need | Lösung | Begründung |
|------|--------|-----------|

| Block-Select | Pointer-Events + pure-functions | CONTEXT D-5. Keine Library. |
| Clipboard | `DataTransfer` + `navigator.clipboard.read/write` | Browser-nativ. CONTEXT D-6 MIME-Type. |
| Virtualisierung (>500 Cells) | DEFERRED | `[ASSUMED]` Performance-Grenze 500 Cells. Realistische Phase-1.2-Modelle (Dummy, Embb, Fertigungsstruktur1) haben unter 1500 Cells. Bosch hat >30k Objekte aber unterschiedlicher Struktur — Performance-Stress-Test extra, nicht in Default-Test-Suite. **Wenn ein User-Use-Case mit großem Modell auftaucht: TanStack Virtual `[ASSUMED]` als Standard-Lösung im React-Ökosystem (zur Phase-1.2-Zeit nicht zu verifizieren ohne Stress-Test).** |

| Context-Menu (Cell-Rechtsklick) | shadcn `<ContextMenu>` (bereits installiert) | Pattern aus PDurchlaufplanViewerDesign (Welle G12). |

### 5.2 Performance-Notes

- **Render-Kostenmodell:** shadcn `<Table>` rendert alle Rows + Cells als DOM-Elemente. Bei 50 Ressourcen × 30 Knoten = 1500 Cells: ~1500 `<td>`-Elemente. Modern-Browser packt das in <50ms initialen Render. Re-Render bei Cell-Edit: nur die eine Cell, weil `useState` lokal im `MatrixCell` ist.
- **Hidden Bottleneck:** `useMemo`-Recompute der Matrix-Mapping-Map bei jeder `allObjects`-Mutation. Wenn `allObjects` ein neues Reference-Objekt ist (immer in Zustand-Immer), läuft der `useMemo` neu. **Mitigation:** Mapping in einer separaten `useMatrixData(rowOids, colOids, ...)`-Hook kapseln, intern mit `useMemo(deps=[rowOids.length, colOids.length, version])` — kein Recompute bei Knoten-Property-Edits, nur bei strukturellen Änderungen.
- **CSS sticky-Performance:** sticky-Header funktioniert nur, wenn `overflow: auto` direkt auf dem Container-Element gesetzt ist. Mit Container-Queries vorsichtig sein (nicht verschachteln).
- **Block-Select Drag-Frequency:** Pointer-Move-Events feuern pro Frame (~60Hz). Selection-Range-Berechnung muss O(1) sein — `computeCellRange(start, end)` ist trivial (Math.min/max), kein Bottleneck.

### 5.3 Test-Pattern (jsdom-tauglich)

- **Sticky-Test:** sticky-CSS ist in jsdom nicht reproduzierbar (kein Layout-Engine). Test via CSS-Class-Presence (`.sticky.top-0.z-10`), nicht via getBoundingClientRect.
- **Block-Select-Test:** pointer-events feuern in jsdom, aber kein realistisches Layout. Tests müssen auf Helper-Functions (`computeCellRange`, `isInRange`) abzielen, nicht auf echtes Drag-Verhalten. **E2E-Test in Playwright** verifiziert das echte Drag-Verhalten.
- **Clipboard-Test:** jsdom hat eingeschränkte `navigator.clipboard`. Test via Mock + Assertion auf MIME-Type + JSON-Payload.

```typescript
const onCellEdit = (row: OBaseObj, columnKey: string, newValue: unknown) => {
  useModelStore.getState().patchObject(row.oid, { [columnKey]: newValue as never });
};

// Phase 1.2 erweitert auf PAssozBeleg-Create-Pfad:
const onCellEdit = (rowRess: OBaseObj, colKnoten: OBaseObj, newStatus: number | null) => {
  const store = useModelStore.getState();
  const existing = findPAssozBeleg(allObjects, colKnoten.oid, rowRess.oid);

  if (newStatus === null) {
    // Cell geleert → Ressource aus PAssozBeleg.m_lRessourcen entfernen
    if (existing) {
      const newList = (existing.attrs.m_lRessourcen as number[]).filter(
        oid => oid !== rowRess.oid,
      );
      if (newList.length === 0) {
        store.deleteObject(existing.oid);
      } else {
        store.patchObject(existing.oid, { m_lRessourcen: newList });
      }
    }
    return;
  }

  if (existing) {
    store.patchObject(existing.oid, { m_eStatus: newStatus });
  } else {
    const newOid = store.createObject("PAssozBeleg", {
      m_lRessourcen: [rowRess.oid],
      m_eStatus: newStatus,
    });
    store.appendSubRef(colKnoten.oid, /* slot=m_lAssozRess */ 0, newOid);
  }
};
```
