### MinIO-Scan auf PRessVerknuepfung-String

Pfad-Schema: `osim-ui-dev/tenants/<tenant_id>/models/<model_uuid>/original.otx`.

Sondierung über alle OTX-Dateien im Bucket `osim-ui-dev`:

```bash
docker exec osim-ui-minio-1 mc ls --recursive local/osim-ui-dev/tenants
# 50 Dateien (33 aus DB + 17 orphaned aus früheren Wellen)
# Pro Datei: mc cat ... | grep -c 'PRessVerknuepfung'
```

**Befund:**

| Metrik | Wert |
|---|---|
| Dateien gescannt | 50 |
| Dateien mit `PRessVerknuepfung`-String | **0** |

**Schlussfolgerung:** Keine Korruption in MinIO. Plausibel, weil:

1. `original.otx` wird beim Upload geschrieben und vom Save-Pfad nie modifiziert (storage_key wird beim Save auf einen neuen `v_*.otx`-Pfad gesetzt). `PRessVerknuepfung`-Korruption könnte nur in EDITED-Versionen entstehen.
2. Welle G16 hat alle korrumpierten EDITED-Versionen via `UPDATE models SET storage_key = original_storage_key` zurückgesetzt — sie sind in der DB nicht mehr referenziert. Die Objekte könnten orphaned im MinIO liegen, aber unser Scan hat alle 50 vorhandenen Dateien geprüft.
3. Die Klass `PRessVerknuepfung` ist im Original C++-OSim2004 nicht definiert — Bestands-Upload-OTX-Dateien können sie daher nicht enthalten.
