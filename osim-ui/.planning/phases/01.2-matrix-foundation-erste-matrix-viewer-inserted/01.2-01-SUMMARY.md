### Task 3: Schema-Sondierung — Wire-Slot-Indizes

**Commit:** `2dd00c8`

Sondierung via `osim_engine.io.otx_reader.parse_otx_file` gegen ein 271-KiB-
Modell aus dem MinIO-Bucket (`tenants/QNxVHEnVE2djdGWqqa0iu9eAv2Ep/models/
e1b0127e-f0d5-41f5-a06e-85bba6d62793/original.otx`, 1480 Objekte). Der
geplante Backend-API-Pfad scheiterte am Setup von `firebase-admin
verify_token` für den Emulator-Token; der direkte OTX-Reader-Pfad liefert
exakt denselben Wire-Stream und war robust.

**Befund:**

- 38 PAssozBeleg, 43 PDpKn-Knoten in 5 Subklassen (PDpKnKonstant,
  PDpKnMengeRuesten, PDpKnRueckKonstant, PDpKnVerteilung,
  PDpKnAlternativVerteilung). 0 direkte PDlplKnoten-Instanzen (Basisklasse
  wird nie direkt instanziiert).
- `PAssozBeleg.sub_refs[0]` ist **immer leer**. Die enthaltene Belegungs-
  Liste lebt in `PRessBelegLList.sub_refs[0]`, deren OID in
  `PAssozBeleg.attrs.m_lRessourcen` als Scalar-Pointer steht.
- `PDpKn*.sub_refs[0]` ist **immer leer**. Die Assoz-Liste lebt in
  `PAssozRessourceLList.sub_refs[0]`, deren OID in
  `PDpKn*.attrs.m_lAssozRess` als Scalar-Pointer steht.

**Konsequenz für Plan 06 Task 2:** `_slotToAttrName` braucht KEINE Einträge
für `PAssozBeleg` / `PDpKn*` (das wäre falsch — würde sub_refs[0] fälschlich
mit dem Scalar-Pointer attrs.m_l* synchronisieren). Stattdessen muss der
Cell-Create-Pfad zwei zusätzliche Schritte machen:

1. **Wrapper-Lazy-Create:** Wenn `parent.attrs.m_lRessourcen === null`, neue
   `PRessBelegLList`-Instanz anlegen (createObject) und ihre OID in
   `parent.attrs.m_lRessourcen` schreiben (patchObject).
2. **appendSubRef auf den Wrapper:** Der Append zielt auf
   `(wrapperOid, 0, childOid)`, nicht auf den PAssozBeleg/PDpKn-Parent.

Die Spec in Welle 1.2-E muss vier Pflicht-Assertions enthalten (Lazy-Create,
Reuse, Delete, Persistenz-Round-Trip), um Korruption à la Welle G16
zu verhindern.

**Doku:** `.planning/phases/01.2-matrix-foundation-erste-matrix-viewer-inserted/01.2-SCHEMA-MAP.md`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocker] Backend-API-Auth-Pfad für Schema-Sondierung umgangen**
- **Found during:** Task 3
- **Issue:** Der Plan-empfohlene Pfad `curl gegen Backend mit Emulator-Token`
  scheiterte mit `401 Invalid token`. Das Backend nutzt
  `firebase-admin verify_token`, das vom Emulator-erzeugten alg:none-Token
  nicht akzeptiert wird (vermutlich fehlt die `FIREBASE_AUTH_EMULATOR_HOST`-
  Env-Var im API-Container für die spezielle verify_token-Path).
- **Fix:** Direkter Zugriff auf den OTX-Wire-Stream via
  `boto3.download_file` (mit den korrekten MinIO-Creds aus dem
  API-Container-Env: `osim_dev` / `osim_dev_password`, NICHT
  `minioadmin`/`minioadmin`) + `osim_engine.io.otx_reader.parse_otx_file`
  direkt im laufenden API-Container. Liefert exakt denselben Wire-Stream wie
  der Backend-API-Call gemacht hätte.
- **Files modified:** keine — reine Sondierung
- **Commit:** Task-3-Doku in `01.2-SCHEMA-MAP.md` (Commit `2dd00c8`)

**2. [Rule 2 - Critical Functionality] Wrapper-Indirektion in SCHEMA-MAP dokumentiert**
- **Found during:** Task 3 Schema-Sondierung
- **Issue:** Die Plan-Annahme „`PAssozBeleg.m_lRessourcen` lebt in `sub_refs[N]`"
  ist falsch. Die Sondierung ergab: `sub_refs[0]` ist immer leer, die Liste
  lebt im verlinkten `PRessBelegLList`-Wrapper. Wäre Plan 06 ohne diese
  Korrektur gestartet, hätte der Cell-Create-Pfad in den falschen Slot
  geschrieben (sub_refs[0] des PAssozBeleg statt im Wrapper) → korrumpierte
  Modelle, identisch zur Phase-1-G16-Problematik.
- **Fix:** SCHEMA-MAP.md dokumentiert die korrekte Wrapper-Indirektion +
  liefert 4 Pflicht-Spec-Assertions (Lazy-Create, Reuse, Delete, Persistenz-
  Round-Trip), die Plan 06 Welle 1.2-E in seine Test-Strategie übernehmen muss.
- **Files modified:** `.planning/phases/01.2-…/01.2-SCHEMA-MAP.md`
- **Commit:** `2dd00c8`

**3. [Rule 3 - Blocker] Worktree-Validierung über docker-cp in Service-Container**
- **Found during:** Task 4 Validierung
- **Issue:** Der Worktree unter
  `.claude/worktrees/agent-af77ee6672b0fd649/portal/` hat kein eigenes
  `node_modules`. `npx tsc` schlug fehl, weil `tsc` nicht auf PATH und
  nicht in einem lokalen node_modules zu finden ist. `npm install` im
  Worktree wäre ~5 Minuten + Disk-Bloat.
- **Fix:** Worktree-Source via `docker cp portal/src
  osim-ui-portal-1:/tmp/wt/src` + Konfig-Files in das laufende
  Frontend-Container-tmp-Verzeichnis kopiert und `node_modules/.bin/tsc` +
  `node_modules/.bin/vitest` von dort gegen die kopierte Source ausgeführt.
  Liefert identische Validierungs-Garantien zu lokalem `npm install`.
- **Files modified:** keine (temporäres /tmp/wt im Container nach Validierung
  wieder entfernt)
- **Commit:** `8902faa` (Validierungs-Output dokumentiert in CLEANUP-LOG.md)
