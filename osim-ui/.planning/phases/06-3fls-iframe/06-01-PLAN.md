<task type="auto" tdd="true">
  <name>Task 2: PostMessageBridge-Klasse mit Origin- und Schema-Validation</name>
  <files>portal/src/embed/PostMessageBridge.ts, portal/src/embed/__tests__/PostMessageBridge.test.ts</files>
  <behavior>
    - `new PostMessageBridge({ allowedOrigins, logger })` registriert genau EINEN `message`-Listener auf `window`.
    - `bridge.on("auth.token", handler)` registriert handler; bei eingehender Message mit passendem type wird handler mit dem validierten Payload aufgerufen.
    - Eingehende Message von nicht-erlaubter Origin: handler NICHT aufgerufen, `logger.warn` mit Origin geloggt.
    - Eingehende Message ohne `type`-Feld oder mit unbekanntem type: verworfen, kein handler-Call, `logger.warn` geloggt.
    - Eingehende Message mit fehlenden Pflichtfeldern (z. B. `auth.token` ohne `token`-String): verworfen.
    - `bridge.send({ type: "ready" })` ruft `window.parent.postMessage(msg, targetOrigin)` mit erster allowedOrigin als targetOrigin (oder explizit gesetzter `parentOrigin`).
    - `bridge.destroy()` entfernt den Listener; nachfolgende Messages triggern keinen handler mehr.
    - Test "Replay-Schutz": Mehrfach gleicher auth.token wird zweimal an handler weitergereicht (Bridge selbst macht keinen Dedup — das ist Verantwortung des Auth-Layers; explizit dokumentiert).
  </behavior>
  <action>
    Implementiere Bridge als Klasse mit private `handlers: Map<string, Set<Handler>>`. Konstruktor: bind `handleMessage = (e: MessageEvent) => {...}`, `window.addEventListener("message", this.handleMessage)`. Validierung: erst `isAllowedOrigin(e.origin)`, dann `typeof e.data === "object" && typeof e.data.type === "string"`, dann type-spezifische Pflichtfeld-Checks (kleines Validator-Objekt pro Inbound-Type — kein zod, manuell ist hier ausreichend). Bei `send`: wenn `parentOrigin` nicht im Konstruktor übergeben, verwende `window.parent` mit `targetOrigin = this.allowedOrigins[0]`. Logger-Interface: `{ warn(msg: string, meta?: object): void; info(...): void }` — Default: `console`. Tests verwenden `vi.fn()` für Handler und Logger; MessageEvents werden via `new MessageEvent("message", { data, origin })` und `window.dispatchEvent(...)` simuliert.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Parent-Window -> Iframe (PostMessage Inbound) | Beliebige Webseite könnte iframe einbetten und auth.token mit gestohlenem Token senden |
| Iframe -> Parent (PostMessage Outbound) | Sensible Daten (z. B. navigate.away) dürfen nicht an fremde Origins gehen |
| URL-Parameter `?embedded=1` | User könnte selbst Standalone-Aufruf mit embedded=1 starten und Auth umgehen wollen |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06.1-01 | Spoofing | PostMessageBridge inbound | mitigate | Strikte Origin-Allowlist (`isAllowedOrigin`) vor jedem Handler-Call; Whitelist aus `VITE_EMBED_ALLOWED_ORIGINS`, niemals `"*"` |
| T-06.1-02 | Tampering | PostMessageBridge inbound | mitigate | Schema-Validation pro Message-Type vor handler-Call; unbekannte/fehlerhafte Messages verworfen und geloggt |
| T-06.1-03 | Information Disclosure | PostMessageBridge outbound | mitigate | `postMessage(msg, targetOrigin)` NIE mit `"*"`, immer explizite Allowlist-Origin |
| T-06.1-04 | Elevation of Privilege | URL-Param `?embedded=1` standalone aufgerufen | accept | Ohne valide PostMessages aus erlaubter Origin kommt nie ein Token an; App bleibt im "waiting for parent"-Zustand; kein Privileg-Gewinn. Dokumentiert. |
| T-06.1-05 | Repudiation | auth.token-Replay | accept | Phase 6.1 macht keinen Dedup; Token-TTL wird in Welle 2 (Backend) auf Server-Seite durchgesetzt. Bridge ist explizit stateless. |
| T-06.1-06 | Denial of Service | Resize-Flood vom Iframe an Parent | mitigate | 100 ms Debounce auf ResizeObserver verhindert Message-Flood |
</threat_model>
