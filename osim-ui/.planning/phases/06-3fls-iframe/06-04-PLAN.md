  <name>Task 1: SecurityHeadersMiddleware und Settings</name>
  <files>app/middleware/security_headers.py, app/core/config.py, main.py, .env.example, tests/middleware/test_security_headers.py</files>
  <behavior>
    - Middleware setzt für JEDE Response folgende Header:
      - `Content-Security-Policy` (oder `Content-Security-Policy-Report-Only` wenn `csp_enforce=False`):
        - `default-src 'self'`
        - `script-src 'self'` (für API: keine inline-Scripts; Portal-Anpassung in Task 2)
        - `style-src 'self'`
        - `img-src 'self' data:`
        - `connect-src 'self'`
        - `frame-ancestors {gespacedte Liste aus settings.csp_frame_ancestors}` — Beispiel: `'self' https://3fls.example.com`
        - falls `csp_report_uri`: `report-uri <uri>`
      - `X-Content-Type-Options: nosniff`
      - `Referrer-Policy: strict-origin-when-cross-origin`
      - `X-Frame-Options: SAMEORIGIN` (Fallback für alte Browser; `frame-ancestors` overridet das in modernen Browsern)
      - `Strict-Transport-Security: max-age=31536000; includeSubDomains` (nur wenn `request.url.scheme == "https"`)
    - Tests: TestClient-Request -> Response enthält alle erwarteten Header mit korrekten Werten.
    - Test "frame_ancestors mit zwei Origins": `settings.csp_frame_ancestors=["'self'","https://3fls.example.com"]` -> Header enthält `frame-ancestors 'self' https://3fls.example.com`.
    - Test "report-only mode": `csp_enforce=False` -> Header heißt `Content-Security-Policy-Report-Only`.
  </behavior>
  <action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Beliebige Origin -> osim-ui-Iframe-Einbettung | Ohne CSP frame-ancestors kann jede Webseite osim-ui einbetten und Clickjacking-Angriffe ausführen |
| Beliebige Origin -> osim-ui-Backend CORS | Browser-cross-origin-Requests sollen nur von 3fls (und osim-ui standalone) erlaubt sein |
| 3fls-Service-Account-JSON | Sensible Credentials; Leak ermöglicht Token-Verifikation-Bypass |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06.4-01 | Tampering | Clickjacking (osim-ui in fremder Iframe) | mitigate | CSP `frame-ancestors` whitelist + X-Frame-Options-Fallback. Verifiziert via evil.html-Browser-Test (Checkpoint Task 4) |
| T-06.4-02 | Information Disclosure | Inline-Script-Injection (XSS) | mitigate | CSP `script-src 'self'` ohne unsafe-inline in Prod. Dev erlaubt unsafe-inline für Vite-HMR mit dokumentierter Begründung |
| T-06.4-03 | Spoofing | Cross-Origin-Request gegen Exchange-Endpoint | mitigate | CORS_ORIGINS-Whitelist; nur 3fls und osim-ui-eigene Origin |
| T-06.4-04 | Information Disclosure | TLS-Downgrade | mitigate | HSTS-Header (1 Jahr, includeSubDomains) in Prod-Responses |
| T-06.4-05 | Information Disclosure | 3fls-SA-Key-Leak | mitigate | Cloud-Run-Secret-Mount statt Image-Bake; Rotation-Schedule im Runbook (90 Tage) |
| T-06.4-06 | Denial of Service | Oversized-Payload an Exchange-Endpoint | mitigate | Pydantic-Validation + (optional) FastAPI Body-Size-Limit via Uvicorn `--limit-max-requests` und Cloud-Run-Default 32 MB. Pentest verifiziert klare Fehler-Response |
| T-06.4-07 | Tampering | CSP-Report-URI selbst attackiert | accept | report-uri ist optional; wenn gesetzt, muss Operator eigenen Endpoint absichern. Nicht im Scope dieser Phase. |
</threat_model>
