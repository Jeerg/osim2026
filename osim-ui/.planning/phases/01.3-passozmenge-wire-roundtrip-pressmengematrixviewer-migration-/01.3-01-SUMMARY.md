Das Audit folgt einem strengen Belegbarkeits-Pattern:

- Jedes persistierte Attribut wird mit einer odh-Zeilennummer belegt (Toleranz ±2 laut Plan-Acceptance-Criteria, tatsächlich exakt: m_iMengeAus=661, m_iMengeEin=735, m_iMengeAbfr=875, m_lMengRess=582).

- Jede Fixture-Option wird mit einem Verdikt versehen (1 = nicht praktikabel, 2 = best match, 3 = overkill).

## Decisions Made

- **PAssozMengeVerbrZwischen hat keine eigenen persistierten Attrs** — erbt komplett von PAssozMengeVerbr. Eigener Loader-Handler trotzdem nötig (Klassenname-Dispatch), aber SCALAR-Tupel identisch.
- **PAssozMenge selbst ist abstract** — kein register_handler / register_writer benötigt.
- **Kein LinkStatusList-Spezialfall** — die vier Subklassen haben keine case-sensitive m_L/m_o-Pointer.
- **m_lMengRess als Scalar-Pointer explizit serialisieren** — analog `_PTagRessWriter`-Pattern (otx_writer.py:892-901), NICHT als blindes `m_l*`-Container-Adopt im Writer.
- **m_lKnoten/m_lOberAssoz optional** — V5-Sim-Slice braucht sie nicht, aber für 1:1-OTX-Treue empfehlenswert. Entscheidung an Plan 01.3-03.
- **Round-Trip-Fixture: Option 2 (konstruiertes pytest)** — bestehende OTX-Fixtures (embb_pre_run, embb_post_run) enthalten KEINE PAssozMenge-Instanzen; Option 1 damit unmöglich. Option 3 (manuelle OTX-Textdatei) zu brüchig. Option 2 nutzt vorhandene `_make_lager`/`_attach_*`-Helpers aus `test_v5_material.py`.
