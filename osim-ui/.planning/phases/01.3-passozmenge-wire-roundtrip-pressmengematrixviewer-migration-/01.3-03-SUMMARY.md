# Phase 01.3 Plan 03: Engine-Writer-Handler PAssozMenge-Familie — Summary

**One-liner:** 5 register_writer-Eintraege fuer die PAssozMenge-Familie in otx_writer.py (Factory _make_passozmenge_writer mit m_sName + variabler Mengen-Scalar + expliziter m_lMengRess-Scalar-Pointer-Serialisierung); 8 Unit-Tests verifizieren Registrierung + Scalars + None-Guard.

## Was wurde getan

### Task 1 — `_make_passozmenge_writer`-Factory + 5 register_writer-Aufrufe

In `engine/src/osim_engine/io/otx_writer.py` direkt nach den drei
`register_writer("PAssozBeleg"|"PAssozRessEnt"|"PAssozELogikEnt")`-Aufrufen
einen Block "Phase 01.3 Welle 2 — PAssozMenge-Familie" eingefuegt
(46 LOC).

**Factory:**

```python
def _make_passozmenge_writer(extra_scalars: tuple[str, ...] = ()):
    scalars = ("m_sName",) + extra_scalars

    class _H(WriterHandler):
        def serialize(self, writer, py, oid):
            props = _serialize_scalars(py, scalars)
            ref = writer.get_oid(getattr(py, "m_lMengRess", None))
            if ref is not None:
                props["m_lMengRess"] = ref
            return props, []

    return _H
```

**5 Registrierungen:**

| Klasse                     | extra_scalars      | SCALARS (effektiv)                  |
|----------------------------|--------------------|--------------------------------------|
| `PAssozMenge`              | `()`               | `("m_sName",)` (abstract — defensiv) |
| `PAssozMengeErzgt`         | `("m_iMengeAus",)` | `("m_sName", "m_iMengeAus")`         |
| `PAssozMengeVerbr`         | `("m_iMengeEin",)` | `("m_sName", "m_iMengeEin")`         |
| `PAssozMengeVerbrZwischen` | `("m_iMengeEin",)` | `("m_sName", "m_iMengeEin")` (erbt)  |
| `PAssozMengeAbfr`          | `("m_iMengeAbfr",)` | `("m_sName", "m_iMengeAbfr")`       |

Pattern-Quelle 1:1 aus AUDIT.md Sektion 2.7 (Attribut-Inventar).

## Race-Condition mit parallelem Plan 02 (zur Dokumentation)

Wahrend der Ausfuehrung lief Plan 02 (otx_loader.py) parallel im selben Repo
ohne Worktree-Isolation. Folgen:

1. Mein RED-Commit `c071bdc test(01.3-03): add failing tests…` wurde durch einen
   parallelen graphify-chore-Commit per `git reset --hard` weggeschoben (siehe Reflog HEAD@{2}: reset moving to HEAD).
2. Das Test-File `engine/tests/unit/io/test_otx_loader.py` von Plan 02 lag teilweise
   im Index meines Working-Trees und wurde versehentlich in meinen ersten Commit
   `339487e` mitgenommen; per `git reset --soft HEAD~1` + `git restore --staged …`
   sauber wieder entfernt.
3. Effektiv liegen Plan 03 jetzt zwei Commits zugrunde:
   - `f85eb28 feat(01.3-03): register 5 PAssozMenge writer handlers in otx_writer.py`
   - `9ddcf9c test(01.3-03): unit tests for 5 PAssozMenge writer handlers`

Konsequenz: separater RED-Test-Commit ging verloren; die Test-Datei wurde im
GREEN-Commit `9ddcf9c` direkt mitgeliefert (die RED-Phase wurde lokal verifiziert
— 8 KeyError-Failures — bevor der Writer-Code committet wurde). TDD-Disziplin
wurde inhaltlich gewahrt, die Git-History zeigt das aber nicht als zwei
getrennte Commits.

Empfehlung fuer Folgepläne: bei parallelen Plänen im selben Repo (kein Worktree)
sollten beide Pläne in getrennte Worktrees gehen, damit Index-Races nicht
auftreten.
