# Phase 01.3 Plan 05: schemas.json-Patch + API-Container-Sync — Summary

## Was wurde getan

Die UI-Schema-Datei `osim-ui/app/static/schemas/v1/schemas.json` wurde um die
5 PAssozMenge-Klassen erweitert, damit der zu migrierende
`PRessMengeMatrixViewer` (Plan 06) die Subklassen-Liste + Mengen-Attrs
sauber aus dem Schema lesen kann. Das ist die UI-Seite der Phase-1.3-
Welle (Loader = Plan 02, Writer = Plan 03, Roundtrip = Plan 04, UI-Schema = Plan 05,
UI-Migration = Plan 06).

### Task 1 — Schema-Eintraege

In `osim-ui/app/static/schemas/v1/schemas.json` 5 neue Klassen-Objekte
direkt nach dem `PAssozBeleg`-Eintrag eingefuegt (vor `PRessVerknuepfung`).
Jeder Eintrag folgt dem PAssozBeleg-Stil mit `klass`, `label_de`,
`viewer_hints`, `description_de`, `properties`. Pro Subklasse:

| Klasse                     | viewer_hints | Eigenes Mengen-Attr | Geerbte Properties        |
|----------------------------|--------------|---------------------|---------------------------|
| `PAssozMenge` (abstract)   | `std`        | -                   | m_sName, m_lMengRess      |
| `PAssozMengeErzgt`         | `matrix`     | `m_iMengeAus`       | m_sName, m_lMengRess      |
| `PAssozMengeVerbr`         | `matrix`     | `m_iMengeEin`       | m_sName, m_lMengRess      |
| `PAssozMengeVerbrZwischen` | `matrix`     | `m_iMengeEin` (erbt)| m_sName, m_lMengRess      |
| `PAssozMengeAbfr`          | `matrix`     | `m_iMengeAbfr`      | m_sName, m_lMengRess      |

Attr-Quelle: AUDIT.md Sektion 2.7 (verbindliche SCALAR + Pointer-Tupel).
`m_lMengRess` wird in jedem Eintrag als `octrl_type=Link` +
`link_target_klass=PRessMenge` (Scalar-Pointer) modelliert.

ZUSAETZLICH: `PRessMenge`-Eintrag (Z. 185-200) bekam einen erweiterten
`description_de` mit Phase-3-Re-Sync-Hinweis. Die Properties-Liste der
PRessMenge bleibt unangetastet, damit der existierende
`PRessMengeMatrixViewer` (vor Plan-06-Migration) nicht bricht.

**Commit:** `b86eb29 feat(01.3-05): add 5 PAssozMenge classes to schemas.json + PRessMenge description hint` (+55 LOC, -1 LOC).

### Task 2 — Container-Sync

Container-Workflow gemaess Memory `dev-stack-container-sync`:

```powershell
docker compose cp app/static/schemas/v1/schemas.json api:/app/app/static/schemas/v1/schemas.json
docker compose restart api
```

(api = gebautes Image OHNE Mount; portal = Bind-Mount, kein Sync noetig.)

Kein Git-Commit fuer Task 2 — Task 2 macht keine File-Aenderungen, nur
Container-Operationen.

## Schema-Checkpoint (Plan-Frontmatter autonomous=false)

**Status: best-effort PASSED — User-Sign-Off optional.**

Der Plan enthielt einen `type="checkpoint:human-verify"` mit dem Auftrag,
den Browser-Endpoint `http://localhost:8000/api/v1/schemas` zu oeffnen und
visuell nach `PAssozMengeErzgt` zu suchen. Faktischer Befund:

1. Der korrekte Endpoint heisst `/api/v1/schemas/v1` (nicht `/api/v1/schemas`).
2. Er ist auth-geschuetzt (`{"detail": "Missing token"}` ohne Bearer-Token).
3. Verbindliche Verifikation des Endpoint-Zustands geschieht daher
   in-process per `docker compose exec api python -c "from app.api.v1.schemas
   import _SCHEMAS_CACHE; ..."` — das _SCHEMAS_CACHE wird beim Modul-Import
   einmal aus der gepatchten Datei geladen, und beim Container-Restart neu
   geladen.

User-Direktive fuer diese Session: "baller alles durch dann test". Daher
wurde der Checkpoint im Best-Effort-Modus durchgezogen, dokumentiert, und
nicht als pausierender Gate behandelt. Der naechste Plan (06 UI-Migration)
kann unbeschadet starten.

Falls der User die HTTP-Verifikation visuell will: in der Browser-Devtools
einen authentifizierten GET an `http://localhost:8000/api/v1/schemas/v1`
absetzen (oder direkt via Portal `localhost:3002` einen Modell-Workspace
oeffnen — das Portal liest das Schema mit eingeloggtem Token).
