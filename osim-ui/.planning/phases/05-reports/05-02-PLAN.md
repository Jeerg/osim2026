  <name>Task 1: Entscheidung Renderer — WeasyPrint vs. Playwright</name>
  <decision>Welcher HTML→PDF-Renderer wird für osim-ui-Reports verwendet?</decision>
  <context>
    Phase-5-PRELIMINARY-PLAN nennt beide Optionen als Discuss-Punkt. Die Entscheidung beeinflusst pyproject.toml-Dependencies, Docker-Image-Größe (Welle-Phase-4-Worker-Image), Tailwind-CSS-Kompatibilität, Chart-Embedding-Strategie und Render-Performance.

    **Hintergrund:**
    - WeasyPrint: Python-natives Tool. Render-Engine basiert auf cairo/pango. Sehr gut für klassische Print-Layouts (CSS 2.1, Paged Media Module). Schwach bei modernem CSS (Grid, Flexbox-Edge-Cases, Tailwind v3-Features). Keine JS-Ausführung. Docker-Image-Footprint klein (~30 MB Extra-Dependencies).
    - Playwright: Headless Chrome. Volle CSS-Modern-Kompatibilität (Tailwind v3 funktioniert 1:1 wie im Browser). JS-Ausführung möglich. Docker-Image-Footprint groß (~250 MB Chrome + Browser-Deps). Render-Performance gut bei warm-gestartetem Browser, schlechter bei Cold-Start.

    **Konkrete osim-ui-Implikationen:**
    - Charts kommen serverseitig als SVG (matplotlib oder vega-lite-CLI). Keine Client-JS-Abhängigkeit nötig → spricht für WeasyPrint.
    - Tailwind-Print-Layout: Standard-Tailwind ist auf Bildschirm optimiert. Für PDF brauchen wir ohnehin einen sparsamen, print-fokussierten CSS-Setup (`@media print`, A4-Margins, Page-Breaks). Tailwind-Grid mit komplexen Subgrid-Features brauchen wir nicht.
    - Phase-4-Worker-Image wird ohnehin größer durch osim-engine + scipy → +250 MB Chrome ist vertretbar, aber nicht ideal.
    - Multi-Tenant-Templates (Plan 03 + 05-05): User-eigene Templates dürfen nur sicheres Subset von HTML/CSS verwenden. WeasyPrint hat hier weniger Angriffsfläche (kein JS).

    **Empfehlung des Planners:** WeasyPrint, sofern ein Prototyp mit dem Standard-Template + 2 SVG-Charts in <10 s rendert und das Layout akzeptabel aussieht. Fallback: Playwright, wenn Tailwind-Print-Setup massive Layout-Probleme zeigt oder Beratungs-Template komplexe moderne CSS-Features braucht.

    **Empfohlenes Vorgehen für die Decision:** 30-Min-Spike: Standard-Template + Dummy-KPIs + 1 SVG-Chart mit WeasyPrint rendern, Output ansehen. Wenn ok → WeasyPrint. Wenn Layout-Bruch → Playwright.
  </context>
  <options>
    <option id="weasyprint">
      <name>WeasyPrint (Python-nativ)</name>
      <pros>
        - Kleine Image-Footprint (~30 MB)
        - Keine JS-Sicherheitsrisiken bei User-Templates
        - Schneller Cold-Start
        - Python-Stack-Konsistenz (keine Node-Toolchain im Worker)
        - Sehr gut für Print-Standardlayouts mit `@page` und `@media print`
      </pros>
      <cons>
        - Tailwind v3-Modern-Features (Grid, Container-Queries) teilweise eingeschränkt
        - Kein JS → falls Charts doch interaktiv sollen, geht das nur nach Plan 06 (Live-Embed)
        - Font-Embedding kann bei Custom-Fonts hakelig werden
      </cons>
    </option>
    <option id="playwright">
      <name>Playwright (Headless Chrome)</name>
      <pros>
        - 100%-Browser-Fidelity (alles was im Live-Viewer funktioniert, funktioniert im PDF)
        - JS-Ausführung möglich (für komplexe Charts, später ggf. Phase-3-Reuse direkt)
        - Tailwind v3 + alle modernen CSS-Features
      </pros>
      <cons>
        - ~250 MB Image-Overhead (Chrome + Deps)
        - Cold-Start 2–4 s (relevant bei On-Demand-Jobs)
        - Sicherheitsrisiko bei User-Templates (JS-Sandboxing nötig)
        - Node.js/Playwright-CLI im Worker-Image — Mehrsprachigkeit
      </cons>
    </option>
    <option id="weasyprint-with-playwright-fallback">
      <name>WeasyPrint als Default + Playwright als opt-in-Fallback pro Template</name>
      <pros>
        - Standardfall billig und sicher
        - Komplexe Templates können bei Bedarf moderne CSS nutzen
      </pros>
      <cons>
        - Zwei Code-Pfade im Worker → mehr Wartung
        - Image enthält beide Renderer → keine Image-Ersparnis
      </cons>
    </option>
  </options>
  <resume-signal>Wähle: `weasyprint`, `playwright`, oder `weasyprint-with-playwright-fallback`. Trage Begründung in `05-02-SUMMARY.md` ein.</resume-signal>
