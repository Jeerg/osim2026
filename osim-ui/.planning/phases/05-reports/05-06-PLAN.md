// Phase-5-Chart-Erweiterung:

interface ChartViewerProps<TData> extends ChildDialogProps<TData> {
  width: number;          // pixel oder pt
  height: number;
  renderMode: 'live' | 'headless';  // unterscheidet Browser-interaktiv vs. PDF-statisch
  exportFormat?: 'svg' | 'png';     // nur relevant bei headless
}

abstract class PChartViewerBase<TData> {
  abstract render(props: ChartViewerProps<TData>): React.ReactElement;
  abstract renderHeadless(data: TData, opts: HeadlessOpts): string;  // gibt SVG-String zurück
}

  <action>
    Entscheidung Headless-Render-Strategie:
    - **Option A (gewählt):** Node.js-Sidecar-Service. Eigener Service im docker-compose (`headless-renderer`) der einen kleinen Express-Endpoint anbietet (`POST /render { chart_type, data, width, height }` → SVG-String). Backend ruft per `httpx` an.
    - **Option B (verworfen):** Python ruft Node-Subprocess pro Chart-Render. Cold-Start jedes Mal teuer, schwer zu skalieren.
    - **Option C (Backlog):** Wechsel zu reinem Python-Renderer (matplotlib bleibt) wenn Sidecar-Komplexität nicht wertgeschätzt wird.

    Backend `app/services/headless_chart_bridge.py`:
    - `class HeadlessChartBridge`:
      - `__init__(self, sidecar_url: str, fallback_renderer: ChartRenderer)`
      - `async def render_chart(chart_type: str, data: dict, *, width: int, height: int) -> str`:
        1. Try: POST sidecar `{chart_type, data, width, height}` → 200 SVG → return
        2. Except (Timeout, ConnectionError, Non-200): fallback auf `self.fallback_renderer` (matplotlib aus Plan 02), structlog-Warning
    - Caching: SVG pro `(run_id, chart_type, hash(data), width, height)` in Redis (aus Plan 02 docker-compose) mit TTL 24h
    - Konfig in `app/core/config.py`: `HEADLESS_RENDERER_URL` (default `http://headless-renderer:3000`)

    Sidecar-Service `portal/src/headless-render-server.ts` (Node.js, einfacher Express):

    - Routing per `chart_type`-Parameter
    - Dockerfile (`Dockerfile.headless-renderer`) mit Node 20 + minimal Deps
    - `docker-compose.yml` ergänzen: Service `headless-renderer` mit Volume-Mount auf `portal/src/viewers`
