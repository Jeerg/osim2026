JSON-Tree Schema (Beispiel-Knoten):
```typescript
interface OtxJsonTree {
  schema_version: "1.0";
  root: OtxJsonNode;
}

interface OtxJsonNode {
  oid: number;              // stabile ID, vom Server vergeben (= Loader-OID)
  klass: string;            // "PAslEinzel", "PDurchlaufplan", "PDpKnKonstant", ...
  name: string;             // m_sName, falls vorhanden, sonst klass + oid
  properties: Record<string, string | number | boolean | null>;  // attr → value (typed)
  children: OtxJsonNode[];  // sub_refs aufgelöst zu nested nodes
  unsupported?: true;       // wenn die Klasse keinen Loader-Handler hat (Passthrough-Skelett)
}
```

Wichtig: `properties` ist TYPED (numerisch als number, bool als boolean), NICHT roh-string wie OTX. Konversion: `app/services/json_tree_service.py` hält die Type-Map (per OTX-Klasse: welches Attribut ist int/float/bool/string).

<risks>
- **Tenant-Table-DDL ohne alembic:** Wenn jemand später eine Spalte zu Model hinzufügt, müssen ALLE existierenden Tenants migriert werden. Mitigation Phase 1: dokumentieren in 01-03-SUMMARY.md, in Phase 4 explizit als TODO "tenant-aware migrations" tracken (Optionen: alembic per-tenant-loop, Postgres-Schema-Templates).
- **OID-Stabilität zwischen GET-Tree und PUT-Tree:** Wenn der Server bei serialize neue OIDs vergibt, kann der Browser-Edit nicht zuverlässig Properties wieder zuordnen. Mitigation: OIDs aus LoadResult.instances umkehren — eindeutige stabile IDs aus dem Loader übernehmen. Browser darf nur OIDs nutzen, die der Server vergeben hat; neue Objekte bekommen vom Browser TEMPORARY-IDs (negativ, z.B. -1, -2), Server vergibt beim PUT echte OIDs zurück (Response enthält id-mapping).
- **Lock-Race-Conditions:** Parallele acquire-Requests von zwei Sessions. Mitigation: UniqueConstraint auf (model_id) als PK + try/except IntegrityError.
- **Storage-Backend-Switch local↔minio:** signed_url-Pfade unterscheiden sich. LocalStorage liefert API-Pfad zurück, der durch Auth-Middleware geschützt sein MUSS — sonst Datenleck. Mitigation: separater `/api/v1/internal-files/{key}`-Endpoint mit eigenem Auth-Check und Tenant-Isolation (key MUSS mit `tenants/{request.tenant_id}/` beginnen).
- **OTX-bytes-Latin-1:** Beim Decode-Encode-Roundtrip via Multipart können Bytes durcheinander geraten. Mitigation: file kommt als bytes über FastAPI (UploadFile.read()), KEIN Decode, direkt an Engine weiterreichen.
</risks>
